## Chapter 0: Git / GitHub とは

### 1. なぜバージョン管理が必要か

ファイルを変更するたびに `report_v1.docx` → `report_v2.docx` → `report_最終.docx` → `report_最終_本当の最終.docx` … となった経験はないでしょうか。これは **バージョン管理（Version Control）** を手動でやっている状態です。

**バージョン管理システム（VCS）** を使うと：

- いつ・誰が・何を・なぜ変えたかを記録できる
- 任意の時点に巻き戻せる
- 複数人が同じファイルを安全に同時編集できる
- 実験的な変更を本線に影響なく試せる

### 2. Git とは

**Git** は 2005 年に Linus Torvalds が Linux カーネル開発のために作った **分散型バージョン管理システム（DVCS）** です。

#### スナップショット方式 vs 差分方式

従来の VCS（CVS・Subversion など）は変更箇所の「差分」を記録します。Git は毎回**プロジェクト全体のスナップショット**を記録します。

| 方式 | イメージ | メリット |
|------|---------|---------|
| **差分方式**（CVS, SVN） | 変更パッチの積み重ね | 保存量が少ない |
| **スナップショット方式**（Git） | 毎回全体の写真を撮る | 任意時点への切替が速い、ブランチが軽量 |

> スナップショットと言っても、変更のないファイルは前回のオブジェクトへのリンクを使うので、実際のディスク消費は効率的です。

#### 分散型 vs 中央集権型

| 方式 | 仕組み | 問題点 |
|------|--------|--------|
| **中央集権型**（SVN） | 唯一のサーバーに全履歴 | サーバー障害で全員作業停止 |
| **分散型**（Git） | 全員が完全な履歴のコピーを持つ | オフラインでも作業可能、サーバー障害に強い |

### 3. GitHub とは

**GitHub** は Git リポジトリをクラウドでホスティングし、チームコラボレーションの機能を追加したプラットフォームです。

Git が「写真の記録技術」なら、GitHub は「写真をみんなで共有・整理するアルバムサービス」です。

| | Git | GitHub |
|--|-----|--------|
| **種類** | ツール（CLI） | Webサービス |
| **役割** | 変更履歴の管理 | リモートホスティング＋コラボレーション |
| **主な機能** | コミット、ブランチ、マージ | Pull Request、Issues、Actions |
| **なくても動くか** | GitHub なしで Git だけ使える | Git なしでは機能しない |

### 4. 関連ツール・サービスの役割分担

Git / GitHub と混同されやすいツール・サービスを整理しておきます。

| ツール | 役割 |
|--------|------|
| **Git** | バージョン管理の実体（ローカルで動く） |
| **GitHub** | Git リポジトリのクラウドホスティング＋コラボレーション |
| **GitHub Actions** | GitHub 上で動く CI/CD 自動化エンジン |
| **GitLab / Bitbucket** | GitHub の競合サービス（同じく Git ベース） |

→ 「Git を使う」＝バージョン管理をする、「GitHub を使う」＝クラウドで共有・協働する、と分けて理解するのがポイントです。

## Chapter 1: 全体像

### 1. Git の 4 つの領域

Git を使うとき、ファイルは常に **4 つの領域のどこか** に存在します。この地図を頭に入れると、コマンドの意味が格段に理解しやすくなります。

```
┌───────────────────────────────────────────────────────┐
│  ローカルマシン                                         │
│                                                       │
│  ┌──────────────┐  git add   ┌────────────────┐       │
│  │  作業ツリー   │ ─────────► │  ステージング  │       │
│  │ (Working     │            │  エリア（Index）│       │
│  │   Tree)      │ ◄───────── │                │       │
│  └──────────────┘  git rm    └────────────────┘       │
│                                      │ git commit     │
│                                      ▼               │
│                             ┌────────────────┐       │
│                             │  ローカルリポ  │       │
│                             │  ジトリ (.git) │       │
│                             └────────────────┘       │
└───────────────────────────────────────────────────────┘
                                      │ git push
                                      ▼
                             ┌────────────────────┐
                             │  リモートリポジトリ │
                             │   (GitHub など)    │
                             └────────────────────┘
                                      │ git fetch / pull
                                      ▼
                              ローカルマシンへ反映
```

| 領域 | 役割 | 対応するコマンド |
|------|------|-----------------|
| **作業ツリー** | 実際に編集するファイル群 | `git status` で状態確認 |
| **ステージングエリア** | 次のコミットに含める変更の一時置き場 | `git add` で追加、`git restore --staged` で取消 |
| **ローカルリポジトリ** | コミット済み履歴の全データ（`.git/` フォルダ） | `git commit` で確定、`git log` で確認 |
| **リモートリポジトリ** | クラウド上の共有リポジトリ（GitHub など） | `git push` で送信、`git pull` で受信 |

### 2. ファイルの状態遷移

作業ツリーのファイルは、Git からみて 4 つの状態を持ちます。

```
Untracked（未追跡）
    │  git add
    ▼
Staged（ステージ済み）
    │  git commit
    ▼
Committed / Unmodified（コミット済み・無変更）
    │  ファイルを編集
    ▼
Modified（変更済み）
    │  git add
    ▼
Staged（ステージ済み）
```

### 3. GitHub の主要機能マップ

GitHub は Git ホスティングの上に、コラボレーション機能を重ねたプラットフォームです。

| カテゴリ | 機能 | 一言で |
|---------|------|-------|
| **コラボレーション** | Pull Request | 変更提案とコードレビュー |
| **コラボレーション** | Issues | タスク・バグ・議論の管理 |
| **コラボレーション** | Projects | カンバン/テーブル形式のプロジェクト管理 |
| **自動化** | GitHub Actions | CI/CD・ワークフロー自動化 |
| **公開・配布** | GitHub Pages | 静的サイトの無料ホスティング |
| **配布** | Packages | npm・Docker イメージ等のパッケージ管理 |
| **開発環境** | Codespaces | クラウド上のフル開発環境 |
| **セキュリティ** | Dependabot / Secret Scanning | 脆弱性・秘密情報の自動検出 |

### 4. 全体の流れ（一人開発 vs チーム開発）

| フェーズ | 一人開発 | チーム開発 |
|---------|---------|----------|
| 変更を記録 | `git add` → `git commit` | 同じ |
| 共有 | `git push` | `git push` → Pull Request 作成 |
| レビュー | 不要 | レビュー → 承認 |
| 統合 | 不要 | Pull Request をマージ |
| 最新化 | `git pull` | `git pull` |

## Chapter 2: Git の内部構造

### 1. Git は「コンテンツアドレス型ファイルシステム」

Git の内部は、ファイルの**内容（コンテンツ）から SHA-1 ハッシュ値を計算**し、それをキーとして保存するデータベースです。

- 内容が同じなら常に同じハッシュ → **重複排除と整合性保証が自動**
- ハッシュが変われば必ず別オブジェクト → **改ざん検知が組み込み**

### 2. 4 種類のオブジェクト

Git の `.git/objects/` には 4 種類のオブジェクトしか存在しません。

| オブジェクト | 対応するもの | 保持する情報 |
|------------|------------|------------|
| **blob** | ファイルの中身 | バイナリデータのみ（ファイル名は持たない） |
| **tree** | ディレクトリ | blob / tree への参照 ＋ ファイル名・パーミッション |
| **commit** | コミット | tree への参照、親 commit への参照、作者・日時・メッセージ |
| **tag** | アノテーションタグ | commit への参照 ＋ タグ名・署名 |

```
commit abc123
  └─ tree def456          ← プロジェクトのスナップショット
       ├─ blob aaa111: src/main.py
       ├─ blob bbb222: README.md
       └─ tree ccc333: tests/
             └─ blob ddd444: tests/test_main.py
```

### 3. ブランチ・HEAD の正体

「ブランチは重い」と思われがちですが、Git のブランチは **単なるコミット SHA-1 を指すポインタファイル（40バイト）** です。

```
.git/
  ├─ HEAD                 ← 現在のブランチを指す（例: ref: refs/heads/main）
  └─ refs/
       └─ heads/
            ├─ main       ← main ブランチの最新 commit の SHA-1
            └─ feature-x  ← feature-x ブランチの最新 commit の SHA-1
```

- **HEAD**: 「今自分がいるブランチ」を指すポインタ
- **ブランチを切る** = 新しいポインタファイルを作るだけ（瞬時・軽量）
- **コミットするとブランチポインタが自動で前進**

### 4. ステージングエリアの役割

ステージングエリア（インデックス）の存在が Git の特徴的な設計です。

```
なぜステージングが必要か？

変更 A ─┐
変更 B ─┤  すべて一緒にコミット（NG）→ 「作業の記録」になってしまう
変更 C ─┘

変更 A ─┐
変更 B ─┘  一緒にステージ → コミット（OK）→ 「機能 X の追加」という意味を持つ
変更 C ──  別にステージ → コミット（OK）→ 「バグ Y の修正」という意味を持つ
```

ステージングを使うことで、**ファイルの編集単位**と**コミットの論理的な単位**を分離できます。

### 5. よくあるアンチパターン

- **`.git/` フォルダを手動編集**: データ破損のリスクがあるため厳禁
- **SHA-1 を覚えようとする**: `git log --oneline` や `HEAD~2` のような参照を使えば十分
- **すべての変更をまとめてステージ**: `git add .` を使いすぎると意味のないコミットになる

## Chapter 3: ブランチとマージ

### 1. ブランチとは何か

ブランチは**本線の開発履歴から分岐した独立した作業空間**です。実験的な変更・新機能・バグ修正を本線（main）に影響なく進められます。

```
main:     A ─── B ─── C ─────────── G
                       \           /
feature-x:              D ─── E ─ F
```

- `D`, `E`, `F` は `feature-x` ブランチ上のコミット
- `main` は `G` のマージコミットで `feature-x` の変更を取り込む

### 2. マージの 2 つの戦略

#### Fast-Forward マージ

`main` が分岐してから **1 件もコミットされていない** 場合、ブランチポインタを前進させるだけで済みます。

```
before:   main: A ─── B
                       \
              feature:  C ─── D

after (fast-forward):
          main: A ─── B ─── C ─── D
```

特徴: **履歴が線形**になる。マージコミットは生まれない。

#### 3-Way マージ

`main` と `feature` が**分岐後にそれぞれコミットが進んでいる**場合、共通祖先・main・feature の 3 点を比較してマージコミットを生成します。

```
before:   main:    A ─── B ─── E
                         \
                 feature:  C ─── D

after (3-way merge):
          main:    A ─── B ─── E ─── M（マージコミット）
                         \          /
                 feature:  C ─── D
```

特徴: **分岐の履歴が残る**。`git log --graph` で可視化できる。

### 3. Rebase: 履歴を線形に保つ

`rebase` は、ブランチの付け根（ベース）を付け替えて、**あたかも最新の main から分岐したかのように履歴を書き換える**操作です。

```
before:   main:    A ─── B ─── E
                         \
                 feature:  C ─── D

after (rebase):
          main:    A ─── B ─── E
                               \
                       feature:  C' ─── D'  ← コミットが別物になる（SHA-1 が変わる）
```

| | Merge | Rebase |
|--|-------|--------|
| 履歴 | 分岐が残る（忠実な記録） | 線形になる（読みやすい） |
| コンフリクト解決 | 1 回（マージコミット時） | コミットごとに対処 |
| 公開済みブランチへの適用 | 安全 | **危険（他者の履歴が壊れる）** |

> **⚠ push 済みのブランチに rebase をかけると、他の人のローカルと履歴が食い違います。** `rebase` はローカルの未 push コミットに限定するのが基本原則です。

### 4. コンフリクト（競合）

同じファイルの同じ箇所を 2 つのブランチが別々に変更した場合、Git は自動解決できず**コンフリクト**を報告します。

```
<<<<<<< HEAD（現在のブランチの内容）
Hello, World!
=======
こんにちは、世界！
>>>>>>> feature-translation（マージしようとしたブランチの内容）
```

解決手順:
1. ファイルを開いて `<<<<`, `====`, `>>>>` の記号を消し、残したい内容に編集
2. `git add <ファイル>` でステージ
3. `git commit` でマージコミットを完成

### 5. よくあるアンチパターン

- **長生きブランチ**: 数週間〜数ヶ月放置するとコンフリクトが爆発する。細かく統合するのが基本
- **`git push --force` の乱用**: 公開済みブランチの履歴を書き換えてチームメンバーの作業を消す危険がある。使う場合は `--force-with-lease` で安全策を
- **マージコミットの多用で履歴が読めなくなる**: PR マージ時に Squash Merge を選ぶと複数コミットを 1 つにまとめて履歴をスッキリ保てる

## Chapter 4: GitHub の主要機能

### 1. Pull Request（PR）

PR は **「このブランチの変更を main に取り込んでほしい」という提案**です。コードレビューとディスカッションの場でもあります。

```
feature-x ブランチで開発
     │
     ▼
git push → GitHub 上に PR を作成
     │
     ▼
レビュワーがコメント・承認
     │
     ▼
Merge ボタンで main に統合
     │
     ▼
feature-x ブランチを削除
```

PR の画面は 4 つのタブで構成されます:

| タブ | 内容 |
|-----|------|
| **Conversation** | PR の説明、コメント、レビューのやりとり |
| **Commits** | このブランチのコミット一覧 |
| **Checks** | GitHub Actions のテスト結果 |
| **Files changed** | 差分（diff）表示とインラインコメント |

#### Draft PR

準備中の変更を早めに共有したい場合、**Draft（下書き）PR** として作成すると「まだレビュー不要」が明示されます。準備ができたら "Ready for review" に変更します。

#### マージ方法の 3 択

| 方法 | 履歴 | 向いているケース |
|------|------|----------------|
| **Merge commit** | 分岐履歴をそのまま保持 | 全コミットを残したい |
| **Squash and merge** | PR の全コミットを 1 つに圧縮 | 作業コミットを整理したい |
| **Rebase and merge** | コミットを線形に接続 | 履歴を線形に保ちたい |

### 2. Issues

Issues は **タスク・バグ・機能要望・議論** を管理するチケット機能です。

- **ラベル**: カテゴリ分類（`bug`, `enhancement`, `documentation` など）
- **Milestone**: リリース単位でのグルーピング
- **Assignee**: 担当者の割り当て
- **PR との紐付け**: PR の説明に `Closes #123` と書くと、PR マージ時に Issue が自動クローズ

### 3. GitHub Actions（CI/CD）

GitHub Actions は **イベント駆動の自動化エンジン**です。コードを push するたびにテストを走らせたり、デプロイを自動化したりできます。

#### 5 つの構成要素

| 要素 | 説明 |
|------|------|
| **ワークフロー** | `.github/workflows/*.yml` に記述する自動化の定義 |
| **イベント** | 実行トリガー（`push`, `pull_request`, `schedule` など） |
| **ジョブ** | 並列・順次実行できる処理の単位 |
| **ステップ** | ジョブ内の個別コマンドまたはアクション |
| **ランナー** | 実行環境（GitHub 提供の Ubuntu/Windows/macOS、またはセルフホスト） |

```yaml
# .github/workflows/ci.yml の例（構造だけ示す）
on: [push, pull_request]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm install
      - run: npm test
```

#### 主な活用シーン

| シーン | 説明 |
|--------|------|
| **CI（継続的インテグレーション）** | PR ごとにテスト・リント・型チェックを自動実行 |
| **CD（継続的デリバリー/デプロイ）** | main へのマージ後に自動デプロイ |
| **定期実行** | 毎晩のバッチ処理、定期レポート生成 |
| **ラベル付け・通知** | PR や Issue への自動ラベル・Slack 通知 |

### 4. その他の主要機能

| 機能 | 概要 |
|------|------|
| **GitHub Pages** | リポジトリの静的ファイルを無料でウェブ公開 |
| **Packages** | npm・Docker・Maven 等のパッケージを GitHub 上でホスティング |
| **Codespaces** | ブラウザから使えるクラウド開発環境（VS Code 互換） |
| **Dependabot** | 依存ライブラリの脆弱性を自動検出→PR で更新提案 |
| **Secret Scanning** | コードに混入した API キーや認証情報を自動検出 |
| **Projects** | Issues / PR をカンバンやテーブルで管理するプロジェクトボード |

### 5. よくあるアンチパターン

- **`main` ブランチへの直 push**: レビューを経ずに変更が入る。Branch Protection Rules で防ぐ
- **PR を放置**: レビューが滞るとコンフリクトが育ち続ける。WIP PR より小さい PR を速くマージする方が健全
- **Secrets をコードに直書きしてから push**: GitHub が自動検出して通知するが、一度 push された秘密情報は git 履歴に残る。必ず環境変数か GitHub Secrets を使う

## Chapter 5: ブランチ戦略

### 1. ブランチ戦略を選ぶ必要性

ブランチをどう使うかをチームで決めておかないと：

- 「いつ何を main にマージするのか」が人によって違う
- 本番環境に未完成の機能が混入する
- コンフリクト解消に追われる

ブランチ戦略は「チームのコード共有ルール」です。代表的な 3 つを比較します。

### 2. Git Flow

2010 年に提唱された、**複数リリースバージョンを同時管理するための戦略**です。

```
main      ──────────────────────────────────────────
              ↑                    ↑
hotfix/    ───┘                    │
                                   │
release/  ──────────── ────────────┘
                    ↑ ↑
develop   ──────────────────────────────────────────
              ↑  ↑  ↑  ↑
feature/  ─────  ─────  ─────  ─────
```

| ブランチ | 役割 |
|---------|------|
| **main** | 本番リリース済みの状態のみ（タグで管理） |
| **develop** | 次回リリースに向けた統合ブランチ |
| **feature/** | 個別機能開発（develop から派生・develop へマージ） |
| **release/** | リリース準備（develop から派生・main + develop へマージ） |
| **hotfix/** | 本番の緊急修正（main から派生・main + develop へマージ） |

**向いているプロジェクト**: バージョン番号付きのライブラリ、複数バージョンを並行サポートするエンタープライズソフトウェア

**注意点**: フローが複雑でツール（`git-flow` コマンド）が必要なほど手順が多い。Web サービスのような「常に1バージョンを継続デプロイ」する場合には過剰。

### 3. GitHub Flow

GitHub が公式に推奨する、**シンプルな 5 ステップのワークフロー**です。

```
main ───────────────────────────────────────────
         ↑              ↑             ↑
feature ─────────────   ──────────    ────────
```

1. **Branch**: `main` から `feature/xxx` を作成
2. **Commit**: こまめにコミットして変更を記録
3. **Pull Request**: レビューのために PR を作成
4. **Review**: コメント・修正・承認
5. **Merge**: 承認後に `main` へマージ → 即デプロイ

**向いているプロジェクト**: Web サービス、SaaS、オープンソース、小〜中規模チーム

**特徴**: `main` は常にデプロイ可能な状態を保つ。シンプルなためオンボーディングコストが低い。

### 4. Trunk-Based Development（TBD）

**全員が毎日 1 回以上、`main`（trunk）に直接インテグレーションする**戦略です。

```
main ──●──●──●──●──●──●──●──
        ↑  ↑     ↑  ↑
short-lived ブランチ（1〜2日以内に統合）
```

- ブランチを長生きさせない（最大 1〜2 日）
- 未完成の機能は **Feature Flag** でコードを隠す
- 強力な CI（コミットごとにテストが通ること）が前提

**向いているプロジェクト**: 継続的デプロイが成熟したチーム、大規模 DevOps 組織

### 5. 戦略の比較と選択基準

| 観点 | Git Flow | GitHub Flow | Trunk-Based |
|------|----------|------------|-------------|
| 複雑さ | 高 | 低 | 中 |
| リリース管理 | 複数バージョン対応 | 常時デプロイ | 常時デプロイ |
| CI/CD 要件 | 低（あれば良い） | 中 | 高（必須） |
| 学習コスト | 高 | 低 | 中 |
| コンフリクト頻度 | 高（長命ブランチ） | 低 | 最低 |
| **向いているチーム** | エンタープライズ / 複数バージョン | スタートアップ / OSS | DevOps 成熟チーム |

→ **迷ったら GitHub Flow から始める**のが現実的です。シンプルで多くのチームに合います。

### 6. よくあるアンチパターン

- **戦略を決めないまま進める**: 暗黙のルールが人によって異なり、本番事故の温床になる
- **Git Flow を使いたくないのに導入**: Web サービスに Git Flow を適用すると `develop` / `release` が形骸化しやすい
- **Trunk-Based を CI なしでやる**: 壊れたコードが常時 main に入るため、CI は必須前提

## Chapter 6: コミットの作法

### 1. コミットは「変更の意図の記録」

コミットはコードの変化だけでなく、**なぜその変更を加えたかという意図**を残す場所です。1 年後の自分や、初めてコードを読む人が履歴から文脈を読み取れるかどうかが品質の基準です。

> 「git blame で誰がこの行を書いたか分かる。でもなぜ書いたかはコミットメッセージにしか残らない」

### 2. Atomic Commit（アトミックコミット）

1 つのコミットに入れる変更は **1 つの論理的な目的** に絞ります。

| ✅ 良い例 | ❌ 悪い例 |
|----------|----------|
| `fix: ログイン時のセッション切れを修正` | `いろいろ直した` |
| `feat: ユーザー一覧にソート機能を追加` | `バグ修正 + ソート + リファクタ` |
| `refactor: user.ts の型定義を整理` | `wip` |

1 コミット = 1 目的にすることで：
- `git bisect` でバグを二分探索できる
- 1 コミットだけ `revert` できる
- コードレビューが読みやすくなる

### 3. Conventional Commits

**Conventional Commits** は、コミットメッセージのフォーマットを標準化する仕様です。CHANGELOG の自動生成やセマンティックバージョニングとの連携に使われます。

```
<type>[optional scope]: <description>

[optional body]

[optional footer(s)]
```

#### type の一覧

| type | 意味 |
|------|------|
| `feat` | 新機能の追加 |
| `fix` | バグ修正 |
| `docs` | ドキュメントのみの変更 |
| `style` | コードの意味に影響しない変更（フォーマット等） |
| `refactor` | バグ修正でも機能追加でもないコード変更 |
| `perf` | パフォーマンス改善 |
| `test` | テストの追加・修正 |
| `chore` | ビルド・依存関係など（本番コードに影響しない） |
| `ci` | CI 設定の変更 |

```
feat(auth): Google OAuth ログインを追加

Firebaseを使わず、OAuth2 ライブラリで直接実装。
既存のセッション管理と統合済み。

Closes #42
```

### 4. コミットメッセージの書き方

**タイトル行（1 行目）** のルール:

- 50 文字以内が理想（GitHub の UI が折り返す）
- 文末にピリオドをつけない
- 命令形で書く: 「修正する」でなく「修正」、`add feature` の形式

**本文（3 行目以降）**:

- なぜその変更が必要だったかを書く
- 「何を変えたか」はコードを見れば分かる。「なぜ変えたか」が価値
- 関連 Issue 番号や PR へのリンクを書く

### 5. よくあるアンチパターン

| パターン | 問題 | 対策 |
|---------|------|------|
| `wip`, `fix`, `あとで直す` | 履歴から何も読み取れない | Conventional Commits を採用 |
| 巨大コミット（100 ファイル変更） | レビュー不能、バグ特定困難 | ステージングを活用して論理単位で分割 |
| 「途中経過」をすべてコミット | 履歴が散らかる | PR マージ時に Squash Merge で整理 |
| `git add .` → コミット | 不要なファイルや機密情報が混入する | `git add -p`（対話的ステージ）で選択する |

## Chapter 7: 次の一歩

### 1. GUI ツール

CLI に慣れていなくても、GUI ツールを使えば視覚的にブランチや差分を確認できます。

| ツール | 特徴 |
|--------|------|
| **GitHub Desktop** | GitHub 公式の GUI クライアント。初心者向けでシンプル |
| **VS Code 組み込み** | エディタと統合。差分確認・ステージ・コミットがエディタ内で完結 |
| **Sourcetree** | Atlassian 製。Git Flow の操作が視覚的にできる |
| **GitKraken** | 豊富なビジュアル。大規模リポジトリ向け |

> GUI は学習の補助として有効ですが、**コマンドを理解してから使うと格段に使いやすくなります**。GUI のボタンが内部で何をしているか分かるようになるからです。

### 2. IDE 統合

主要 IDE には Git 連携が組み込まれています。

| IDE / エディタ | 機能 |
|---------------|------|
| **VS Code** | Source Control パネル、差分表示、GitLens 拡張 |
| **JetBrains（IntelliJ 等）** | Git ツールウィンドウ、インラインコミット履歴 |
| **Vim / Neovim** | fugitive.vim プラグイン |

### 3. `.gitignore` の活用

追跡したくないファイル（ビルド成果物、秘密情報、OS 生成ファイル等）は `.gitignore` に記述します。

```
# 依存関係
node_modules/
__pycache__/

# ビルド成果物
dist/
*.o

# 環境変数・秘密情報
.env
.env.local

# OS・エディタ
.DS_Store
.vscode/
```

`gitignore.io` で言語・フレームワーク別の雛形を生成できます（`https://www.toptal.com/developers/gitignore`）。

### 4. 便利なコマンド早見表

| 目的 | コマンド |
|------|---------|
| 変更をいったん退避 | `git stash` / `git stash pop` |
| 特定コミットを cherry-pick | `git cherry-pick <SHA>` |
| コミットを取り消す（履歴保持）| `git revert <SHA>` |
| バグを二分探索 | `git bisect start` / `git bisect good` / `git bisect bad` |
| 対話的にコミットを整理 | `git rebase -i HEAD~N` |
| 変更した行の履歴を見る | `git log -p <ファイル>` |
| 行の最終変更コミット | `git blame <ファイル>` |
