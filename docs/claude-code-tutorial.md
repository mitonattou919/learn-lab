# Claude Code チュートリアル

> Claude Code を初めて触る人向けのハンズオン教材。8章を順に進めれば基本操作と最低限の作法が身につきます。
> もっと深く知りたくなったら → `claude-code-best-practices.md`

---

## Chapter 0: Claude Code とは

### AIコーディングエージェント

ChatGPT や Claude.ai は **会話だけ** のAIです。コードを提案してくれても、実際にファイルに反映するのは人間の仕事でした。

**Claude Code** は一歩進んで、ターミナル上で動き、自分でファイルを読み書きしたりコマンドを実行したりできます。

```
人間: 「README を日本語に翻訳して」
  ↓
Claude Code: README.md を読む → 翻訳する → ファイルを書き換える → 完了報告
```

### ChatGPT との違い

| | ChatGPT / Claude.ai | Claude Code |
|--|---------------------|-------------|
| 動作環境 | ブラウザ | ターミナル |
| ファイル操作 | できない | 直接読み書き |
| コマンド実行 | できない | できる |
| コンテキスト | 会話の中だけ | プロジェクト全体 |

### 「優秀なジュニアエンジニア」だと思って付き合う

- 指示が明確であれば確実にこなす
- 曖昧な指示には曖昧な結果が返る
- 最終的な判断と責任は人間が持つ

## Chapter 1: インストールと初回ハロー体験

### 1. 必要な環境

- macOS / Linux / Windows（WSL2推奨）
- Node.js v18 以上
- Anthropic アカウント（claude.ai）

```bash
node --version   # v18 以上を確認
```

### 2. インストール

```bash
npm install -g @anthropic-ai/claude-code
claude --version
```

### 3. 初回起動と認証

```bash
claude
```

ブラウザが開くので claude.ai でログインして権限許可。これだけで認証完了です。

> **うまくいかない時**: `claude: command not found` が出たら `npm config get prefix` で出るパスが `$PATH` に入っているか確認。

### 4. 5分で成功体験

練習用ディレクトリを作って遊んでみましょう。

```bash
mkdir claude-practice && cd claude-practice
echo "console.log('hello')" > index.js
claude
```

起動したら、こう聞いてみてください：

```
> このディレクトリにあるファイルを教えて
> index.js を読んで何をしているか説明して
> index.js を「こんにちは、世界！」と出力するように修正して
> node index.js を実行して結果を見せて
```

これが Claude Code の基本動作です。**「読む・編集する・実行する」** を自然言語で指示するだけ。

### 5. 終了

```
> /exit
```

または `Ctrl+D`。

## Chapter 2: 基本操作

### 1. `@` でファイルを参照

最も使う書き方です。`@パス` でファイルの内容を Claude に渡せます。

```
> @src/index.ts を読んでバグがないか確認して
> @package.json を見て利用可能なコマンドを教えて
```

複数指定もOK：

```
> @src/api.ts と @src/types.ts を読んで型の整合性を確認して
```

### 2. エラーをそのまま貼る

エラーメッセージはコピペで投げるのが一番早い。

```
> 以下のエラーが出た。原因と修正方法を教えて：
>
> Error: ENOENT: no such file or directory, open '/app/config.json'
>     at loadConfig (src/config.ts:12:18)
```

### 3. 必須スラッシュコマンド

| コマンド | 用途 |
|---------|------|
| `/help` | ヘルプ |
| `/clear` | 会話履歴リセット（タスク切替時に） |
| `/login` | 再認証 |
| `/model` | モデル変更 |
| `/permissions` | 許可ルール確認 |
| `/exit` | 終了 |

### 4. 許可プロンプト

ファイル編集やコマンド実行の前に、Claude が確認してきます。

```
Allow? [y/n/always]
```

| 選択 | 意味 |
|------|------|
| `y` | 今回だけ許可 |
| `n` | 拒否 |
| `always` | このセッション中は常に許可 |

> **基本方針**: 内容を読んでから選ぶ。同じコマンドを何度も実行する場合のみ `always`。

### 5. 練習課題

`claude-practice/` の中で：

1. `hello.py` を「Hello, World!」と出力する Python スクリプトとして作らせる
2. `python hello.py` を実行させる
3. 出力を「ようこそ、Python の世界へ」に変更させる

## Chapter 3: 良い指示の書き方

Claude Code は賢いですが **テレパシーは使えません**。指示の質が結果の質を決めます。

### 1. 4要素を意識する

| 要素 | 質問 | 例 |
|------|------|-----|
| **対象** | 何を？ | `src/api.ts の fetchUser 関数` |
| **目的** | なぜ？ | `本番のデバッグをしやすくしたい` |
| **制約** | どうやって？ | `外部ライブラリは使わず標準APIだけで` |
| **完了条件** | どうなれば終わり？ | `pnpm test が全パスしたら完了` |

### Before / After

| 悪い例 | 良い例 |
|--------|--------|
| `直して` | `src/api.ts の fetchUser を、ユーザー不在時に undefined を返す代わりに NotFoundError を投げるよう修正して` |
| `バリデーション追加して` | `src/user.ts の createUser に、email が空文字または @ を含まない場合はエラーを投げるバリデーションを追加して` |
| `ログ処理を改善して` | `src/logger.ts の log() を、本番環境ではスタックトレースも出すように改善して` |

### 2. ⭐ 検証(Verify)手段を必ず渡す

これが**最重要原則**です。Claude が自己検証できるようにすると、結果が劇的に良くなります。

| 種類 | 渡し方 |
|------|--------|
| ロジック | テストケース（入力→期待出力）を書く |
| UI | スクショを貼って差分比較させる |
| バグ修正 | エラーメッセージと再現手順を貼る |
| ビルド | `pnpm build` のような確認コマンドを指定 |

```
validateEmail 関数を実装して。

テストケース:
  user@example.com → true
  invalid → false
  user@.com → false

実装後にテストを実行して全て通ることを確認して。
```

> **検証手段が無いものは出さない**。「動いてるっぽい」は本番で壊れます。

### 3. 練習課題

以下の指示を4要素＋検証手段を入れて書き直してみてください：

```
元の指示: 「日付フォーマット関数を作って」
```

## Chapter 4: Plan Mode と段階的な進め方

### 1. Plan Mode で「読むだけ」モードに

複数ファイルを触る作業や、初めて読むコードベースを修正する時は、いきなり書かせず**まず計画を立てさせる**のが安全です。

`Shift+Tab` を押すと **Plan Mode** に切り替わります。このモードでは Claude はファイルを編集しません、読んで提案するだけ。

```
[Plan Mode]
> src/auth/ を読んで、Google OAuth を追加するなら
> どのファイルをどう変える必要があるか計画を立てて
```

計画に納得したら `Shift+Tab` で通常モードに戻して実装させます。

### 2. いつ Plan Mode を使うか

| 使う | 使わない |
|------|---------|
| 複数ファイルにまたがる変更 | タイポ修正 |
| 初めて触るコード | ログ1行追加 |
| 設計判断が含まれる | 変数リネーム |
| 不可逆な操作（DB、削除） | 1文で diff を説明できる修正 |

### 3. タスク粒度別の指示

#### 小: 一度に指示

```
> @src/index.ts の23行目、typo「recieve」を「receive」に直して
```

#### 中: ゴールベースで指示

「手順」でなく「達成したい状態」を伝えます。

```
> @src/api.ts 内の全関数にエラーハンドリングを追加して。
> 追加後はテストを実行して既存テストが全パスすることを確認して。
```

#### 大: 段階的に進める

```
# Step 1（調査）
> src/ 全体を読んでテストカバレッジが低いファイルを特定して

# Step 2（計画）— 確認してから次へ
> どのファイルにどんなテストを追加するか方針を提案して。実装はまだしない

# Step 3（実行）
> 提案した方針でテストを追加して。1ファイルごとに変更内容を報告して
```

### 4. 途中で止める・方向転換する

| 操作 | 用途 |
|------|------|
| `Esc` | 進行中の処理を中断（履歴は残る） |
| `Esc Esc` または `/rewind` | チェックポイントに巻き戻し |
| `Undo that` と入力 | 直前の変更を取り消し |

間違った方向に進んでいると気づいたら **早めに止める**。「まずいけど途中まで進んだし」と続けるとロールバックが大変になります。

### 5. 練習課題

Plan Mode で `claude-practice/` を分析させ、「このプロジェクトをもう少し本格的にするなら何を追加すべきか」を提案させてみてください。実装はしません、提案だけ。

## Chapter 5: CLAUDE.md でプロジェクト知識を持たせる

### 1. CLAUDE.md とは

プロジェクトルートに置くと、**Claude Code が起動時に自動で読み込む**設定ファイルです。「毎回説明する手間」を省けます。

### 2. `/init` で雛形を作る

プロジェクトのルートで：

```
> /init
```

Claude がコードベースを分析して CLAUDE.md の叩き台を作ってくれます。あとは育てるだけ。

### 3. 配置場所

| パス | 用途 |
|------|------|
| `~/.claude/CLAUDE.md` | 全プロジェクト共通の個人ルール |
| `./CLAUDE.md` | プロジェクトのチーム共通設定（git管理） |
| `./CLAUDE.local.md` | 個人メモ（`.gitignore` に追加） |

### 4. 書く例

```markdown
# プロジェクト名

### 技術スタック
- TypeScript + Node.js 20
- フレームワーク: Hono
- テスト: Vitest
- パッケージマネージャ: pnpm

### よく使うコマンド
- `pnpm dev`: 開発サーバー起動
- `pnpm test`: テスト実行
- `pnpm typecheck`: 型チェック

### コーディングルール
- コメントは日本語
- any 型の使用は禁止

### 注意事項
- DB マイグレーションは必ず確認を取る
- console.log を本番コードに残さない
```

### 5. 書くこと / 書かないこと

| ✅ 書く | ❌ 書かない |
|--------|-----------|
| Claude が推測できないコマンド | コードを読めば分かること |
| プロジェクト固有のルール | 言語標準の慣習 |
| 環境変数などの落とし穴 | 一般論（「クリーンなコードを書く」） |

### 6. 鉄則: 長くしすぎない

**長すぎる CLAUDE.md は無視されます**。各行に「これを消したら Claude がミスるか？」を問うて、答えが No なら削る。

Claude が CLAUDE.md に書いてあるルールを破る場合、たいてい **長すぎ** が原因です。

### 7. 練習課題

`claude-practice/` に CLAUDE.md を作り、`claude` を再起動。「このプロジェクトのルールを教えて」と聞いて、ちゃんと読まれているか確認してください。

## Chapter 6: セッション管理

### 1. `/clear` でコンテキストをリセット

長い会話を続けると Claude が初期の指示を「忘れる」ことがあります。これは **コンテキストウィンドウ**（Claude が一度に覚えていられる範囲）が埋まるためです。

**無関係なタスクに移る時は `/clear` する**のが鉄則。

```
> /clear
```

### 2. セッションを再開する

会話履歴はローカルに保存されているので、ターミナルを閉じても続きから戻れます。

```bash
claude --continue    # 直近のセッションを再開
claude --resume      # 一覧から選んで再開
```

長期作業は `/rename oauth-migration` のように名前を付けておくと、後から見つけやすくなります。

### 3. `/rewind` で巻き戻す

Claude の各操作前に**自動でチェックポイント**が作られています。`Esc Esc` または `/rewind` で：

- 会話だけ戻す
- コードだけ戻す
- 両方戻す

から選べます。「とりあえず試してダメなら戻す」攻めの進め方ができます。

> ⚠ Claude が行った変更のみが対象。git の代わりにはなりません。

### 4. 同じ問題で2回失敗したら

同じことで何度も修正させていると、文脈が **失敗履歴で汚染** されて精度が落ちます。

→ `/clear` して、学んだことを盛り込んだ良いプロンプトで再開する方が早い。

### 5. 練習課題

1. `claude-practice/` で何かタスクをやる
2. `/exit` で終了
3. 翌日 `claude --continue` で再開し、続きから話せることを確認

## Chapter 7: 次の一歩

ここまでで Claude Code の基本は身につきました。実務に組み込んでいくには次のトピックがあります。

### もっと深く学ぶ

→ **`claude-code-best-practices.md`** に進んでください。以下が学べます：

- **Auto Mode / Sandbox** で承認プロンプトを減らす
- **Skills / Subagents / Hooks / MCP / Plugins** の拡張機構
- **並列セッション** と **Writer/Reviewer パターン**
- **非インタラクティブモード** (`claude -p`) で CI に組み込む
- **Fan-out** で大量ファイルを一括処理
- チーム運用設計と失敗パターン集

### 直感を育てる

ガイドのパターンは出発点です。使い続けるうちに：

- いつ Plan Mode を使うか
- いつ `/clear` するか
- どの粒度で指示すれば伝わるか

が体感的に分かってきます。**うまくいった時と詰まった時の両方で「なぜそうなったか」を振り返る** のが上達の近道です。
