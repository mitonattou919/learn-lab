## Chapter 0: Docker とは

### 1. 「環境差異」という古くて深い問題

「自分のマシンでは動くのに本番で動かない」——これはソフトウェア開発の世界で長年語り継がれる苦悩です。原因はほぼ決まっています。OSのバージョン、ランタイムのバージョン、インストール済みライブラリ、環境変数……開発・テスト・本番の環境がわずかでも違うと、動作が変わります。

**Docker** はこの問題を「アプリケーションとその実行環境をまるごとパッケージ化して持ち運ぶ」という発想で解決します。

```
開発者のMacBook  →  CIサーバー  →  本番Linux  
                                     どこでも同じ動作
      ↑──────────────────────────────↑
         Docker イメージという「箱」に詰める
```

### 2. Docker とは何か

**Docker** は、アプリケーションを **コンテナ（Container）** という軽量な実行単位にパッケージ化・配布・実行するためのオープンプラットフォームです。2013 年に Solomon Hykes が中心となって開発し、Docker 社（現 Mirantis に一部事業譲渡）が公開しました。

メタファーとして覚えると便利です。**Docker は「標準化されたコンテナ（輸送用コンテナ）」をソフトウェアの世界に持ち込んだもの**です。輸送用コンテナが中身に関わらず同じ形・同じ規格で船・トレーラー・クレーンに対応できるように、Docker コンテナはどのホスト OS・どのクラウドでも同じように実行できます。

### 3. 仮想マシン（VM）との役割分担

Docker を学ぶとき最初に整理すべきなのが「VM との違い」です。どちらも「環境を分離する」技術ですが、仮想化するレイヤーが根本的に異なります。

| 観点 | 仮想マシン（VM） | Docker コンテナ |
|------|----------------|----------------|
| 仮想化レイヤー | ハードウェアレベル（Hypervisor） | OS レベル（カーネル共有） |
| ゲスト OS | 各 VM が完全な OS を持つ（数 GB） | ホスト OS のカーネルを共有（数 MB） |
| 起動時間 | 数分 | 数秒以下 |
| リソース消費 | 重い（CPU・メモリを多く占有） | 軽量・高密度 |
| 分離レベル | 強固（完全なハードウェア分離） | プロセスレベル（名前空間で分離） |
| 向いている場面 | 異なる OS が必要・強い分離が必要 | 同じ OS 系・高密度デプロイ・CI |

> **どちらを使うべきか**: 相互排他ではありません。本番では VM（またはベアメタル）の上で Docker コンテナを動かすのが一般的な構成です。

### 4. Docker が解決する 3 つの課題

| 課題 | Docker の解決策 |
|------|---------------|
| **環境差異** | イメージで実行環境ごとパッケージ化→どこでも同じ動作 |
| **セットアップコスト** | `docker run` 一発で依存込みの環境が起動 |
| **デプロイの一貫性** | レジストリ経由でビルド済みイメージを配布→コードとインフラを一体管理 |

## Chapter 1: 全体像・主要コンポーネント

### 1. Docker を構成する 5 つの要素

Docker は単一のツールではなく、複数のコンポーネントが協調して動くシステムです。全体像を先に把握しておくと、個々の概念がどこに位置するかが分かりやすくなります。

```
┌──────────────────────────────────────────────────────────┐
│  開発者                                                   │
│    ↓  docker build / run / push ...                      │
│  ┌──────────────┐                                        │
│  │  Docker CLI  │  ←── コマンドラインインターフェース      │
│  └──────┬───────┘                                        │
│         │ REST API                                       │
│  ┌──────▼───────────────────────────────┐               │
│  │  Docker Daemon (dockerd)             │               │
│  │  ┌──────────┐  ┌──────────────────┐  │               │
│  │  │  Images  │  │   Containers     │  │               │
│  │  ├──────────┤  ├──────────────────┤  │               │
│  │  │ Networks │  │    Volumes       │  │               │
│  │  └──────────┘  └──────────────────┘  │               │
│  └──────────────────────────────────────┘               │
│         ↕ push / pull                                   │
│  ┌──────────────┐                                        │
│  │  Registry    │  ←── Docker Hub / プライベートレジストリ │
│  └──────────────┘                                        │
└──────────────────────────────────────────────────────────┘
```

### 2. 主要コンポーネントの役割

| コンポーネント | 役割 | たとえるなら |
|--------------|------|------------|
| **Docker CLI** | ユーザーが Daemon へ指示を送るコマンドラインツール | 電話の送受話器 |
| **Docker Daemon (`dockerd`)** | ホストで常駐しコンテナ・イメージ・ネットワーク・ボリュームを管理するサーバープロセス | 管制塔 |
| **containerd** | Daemon の下で実際のコンテナ起動・停止を担う低レベルランタイム | 滑走路 |
| **Docker Desktop** | Windows / macOS 向けの GUI 付き統合ツール。Daemon・CLI・Compose を同梱 | オールインワン空港 |
| **Registry** | イメージを保存・配布するサーバー。Docker Hub が標準 | 荷物の物流センター |

> **Docker Desktop と Docker Engine の違い**: Docker Engine（Daemon + CLI）はLinuxネイティブのコアです。Docker Desktop はその上にGUI・拡張機能・組み込みKubernetesを乗せたWindows/macOS向けラッパーです。大企業での商用利用は有償サブスクリプションが必要です（2022年〜）。

### 3. ワークフローの全体像

Docker を使った開発〜デプロイの流れは、大きく 4 ステップです。

```
① 記述        ② ビルド       ③ 配布         ④ 実行
Dockerfile  →  Image      →  Registry   →  Container
（設計図）      （テンプレート）  （倉庫）        （実行インスタンス）
```

| ステップ | コマンド例 | 何が起きるか |
|---------|-----------|------------|
| ① Dockerfile を書く | —（エディタで編集） | イメージの構成手順を定義 |
| ② イメージをビルド | `docker build -t myapp:1.0 .` | Dockerfile からイメージを作成 |
| ③ レジストリへ送信 | `docker push myapp:1.0` | 他のマシンからも取得可能になる |
| ④ コンテナとして起動 | `docker run myapp:1.0` | イメージを元にコンテナが起動 |

## Chapter 2: Image（イメージ）

### 1. イメージとは「実行環境の設計図」

**Docker イメージ**は、コンテナを起動するための読み取り専用のテンプレートです。OS のファイルシステム・アプリのコード・依存ライブラリ・設定ファイルなど、実行に必要なすべてが含まれています。

オブジェクト指向で例えるなら、「イメージ = クラス、コンテナ = インスタンス」の関係に近いです。1 つのイメージから何個でも独立したコンテナを起動できます。

```
Image（クラス）
  ├─ OS ファイルシステム（例: Ubuntu 22.04）
  ├─ ランタイム（例: Node.js 20）
  ├─ アプリコード
  └─ 環境変数・起動コマンドの定義

     ↓ docker run（何度でも）

Container A  Container B  Container C  ...（それぞれ独立）
```

### 2. レイヤー構造：差分だけを積み重ねる

イメージの最大の特徴が **レイヤー構造**です。Dockerfile の各命令（`RUN`・`COPY`・`ADD` など）が 1 つの読み取り専用レイヤーを生成し、それを積み重ねてイメージが構成されます。

```
┌──────────────────────────────┐  ← Layer 4: アプリコードをコピー（COPY）
├──────────────────────────────┤  ← Layer 3: 依存ライブラリインストール（RUN npm install）
├──────────────────────────────┤  ← Layer 2: Node.js インストール（RUN apt-get install）
├──────────────────────────────┤  ← Layer 1: ベースイメージ（FROM ubuntu:22.04）
└──────────────────────────────┘
```

**レイヤーキャッシュ**が重要な概念です。一度ビルドしたレイヤーは変更がない限りキャッシュされ再利用されます。Dockerfile の命令順序を工夫することで、ビルド時間を大幅に短縮できます。

| 変更したレイヤー | 再ビルドが必要なレイヤー |
|---------------|----------------------|
| Layer 1（ベース） | Layer 1 〜 4 すべて |
| Layer 3（依存ライブラリ） | Layer 3・4 のみ |
| Layer 4（アプリコード） | Layer 4 のみ（最速）|

→ 「変わりにくいレイヤーを下に、変わりやすいレイヤーを上に」配置するのがセオリーです（→ Chapter 8 のアンチパターンでも詳述）。

### 3. Dockerfile：イメージのレシピ

**Dockerfile** はイメージのビルド手順をテキストで記述したファイルです。代表的な命令を紹介します。

| 命令 | 役割 | 例 |
|-----|------|---|
| `FROM` | ベースイメージを指定 | `FROM node:20-alpine` |
| `RUN` | コマンドを実行してレイヤーを作る | `RUN npm install` |
| `COPY` | ホストからファイルをコピー | `COPY . /app` |
| `WORKDIR` | 作業ディレクトリを設定 | `WORKDIR /app` |
| `ENV` | 環境変数を設定 | `ENV NODE_ENV=production` |
| `EXPOSE` | 公開ポートを宣言（ドキュメント的な意味） | `EXPOSE 3000` |
| `CMD` | コンテナ起動時のデフォルトコマンド | `CMD ["node", "server.js"]` |

> ⚠ **よくあるアンチパターン**: `FROM ubuntu:latest` のように `latest` タグを使うと、ビルドするたびにベースイメージが変わる可能性があります。`FROM ubuntu:22.04` のようにバージョンを固定するのが推奨です。

### 4. イメージの命名規則

イメージは `[レジストリ/]リポジトリ名:タグ` の形式で識別されます。

```
docker.io/library/nginx:1.25.3
   ↑         ↑       ↑
レジストリ  リポジトリ  タグ（バージョン）

省略すると → nginx （= docker.io/library/nginx:latest）
```

## Chapter 3: Container（コンテナ）

### 1. コンテナとは「イメージの実行インスタンス」

**コンテナ**はイメージを実際に起動したものです。イメージ（読み取り専用）の上に薄い**書き込み可能レイヤー**が追加され、独立したプロセス・ファイルシステム・ネットワーク空間を持って動作します。

```
┌──────────────────────────────────┐
│  Container（書き込み可能レイヤー） │ ← ここだけ変更可能（コンテナ固有）
├──────────────────────────────────┤
│  Image Layer 4                   │
├──────────────────────────────────┤  ← 読み取り専用（イメージ）
│  Image Layer 3                   │
├──────────────────────────────────┤
│  Image Layer 2                   │
├──────────────────────────────────┤
│  Image Layer 1（ベース）          │
└──────────────────────────────────┘
```

> コンテナを停止・削除しても、元のイメージは変化しません。書き込み可能レイヤーに加えた変更はコンテナと一緒に消えます。データを永続化したい場合は Volume（Chapter 5）を使います。

### 2. コンテナのライフサイクル

コンテナは「作成→起動→停止→削除」という状態を持ちます。

```
 docker create          docker start
    ↓                      ↓
[Created]  ──────────►  [Running]  ────────►  [Exited]
                                      ↑            │
                         docker stop / exit        │ docker rm
                                                   ▼
                                              （削除）
```

| 状態 | 説明 |
|------|------|
| **Created** | イメージからコンテナが作成済み。まだ起動していない |
| **Running** | プロセスが実行中 |
| **Paused** | プロセスが一時停止中（`docker pause`） |
| **Exited** | プロセスが終了した。コンテナ自体は残っている |
| **Removed** | `docker rm` で削除済み |

### 3. コンテナの分離の仕組み

コンテナは VM のように完全なゲスト OS を持つのではなく、Linux カーネルの 2 つの機能で分離を実現しています。

| 機能 | 役割 |
|------|------|
| **Namespace（名前空間）** | プロセス・ネットワーク・ファイルシステム・ユーザーなどをコンテナごとに独立した「見え方」にする |
| **cgroup（コントロールグループ）** | CPU・メモリ・I/O などのリソース使用量をコンテナごとに制限・計測する |

この仕組みにより、コンテナは「ホストから見ると 1 プロセス、コンテナ内から見ると独立した環境」として動作します。

### 4. コンテナ設計の原則

| 原則 | 概要 |
|------|------|
| **1 コンテナ 1 プロセス** | Web サーバー・DB・アプリは別コンテナに分ける。複数プロセスを詰め込まない |
| **イミュータブル（不変）** | 動いているコンテナを直接変更しない。変更はイメージを作り直してデプロイ |
| **ステートレス** | コンテナ内にデータを持たない。状態は Volume や外部 DB に逃がす |

## Chapter 4: Network（ネットワーク）

### 1. コンテナはデフォルトで「孤島」

Docker コンテナはデフォルトで互いに隔離されたネットワーク空間を持ちます。Web サーバーとデータベースが別コンテナにある場合、何も設定しなければ通信できません。**Docker ネットワーク**はこの「コンテナ間通信」と「外部公開」を制御する仮想ネットワーク機能です。

### 2. ネットワークドライバの種類

Docker は用途に応じて複数のネットワークドライバを提供しています。

| ドライバ | 用途 | 特徴 |
|---------|------|------|
| **bridge**（デフォルト） | 単一ホスト上のコンテナ間通信 | 仮想スイッチを作成。ポートマッピングで外部公開 |
| **host** | コンテナをホストのネットワークに直結 | 分離なし・最高速。Linux 専用 |
| **none** | ネットワーク完全無効 | 完全に孤立したコンテナ |
| **overlay** | 複数ホスト間のコンテナ通信 | Docker Swarm / Kubernetes で使用 |
| **macvlan** | コンテナに MAC アドレスを割り当て | ネットワーク上で独立した物理機器に見せる |

### 3. bridge ネットワークの仕組み

最もよく使う `bridge` ドライバの動作を理解しておきましょう。

```
ホスト OS
┌─────────────────────────────────────────────┐
│  ┌──────────────┐    ┌──────────────┐       │
│  │  Container A │    │  Container B │       │
│  │  172.17.0.2  │    │  172.17.0.3  │       │
│  └──────┬───────┘    └──────┬───────┘       │
│         └──────────┬─────────┘              │
│               docker0（仮想ブリッジ）         │
│               172.17.0.1                    │
│                    │                        │
│               ホストのNIC                   │
└─────────────────────────────────────────────┘
                     │
                  インターネット
```

**ポートマッピング**でコンテナ内ポートを外部に公開します。

```bash
docker run -p 8080:3000 myapp
#              ↑     ↑
#          ホスト  コンテナ内
```

### 4. ユーザー定義ネットワークと DNS 解決

デフォルトの `bridge` ネットワークではコンテナ名で通信できませんが、**ユーザー定義ネットワーク**を作るとコンテナ名が自動的に DNS で解決されます。

| 比較 | デフォルト bridge | ユーザー定義ネットワーク |
|------|-----------------|----------------------|
| コンテナ名での通信 | ❌ 不可（IP 直指定が必要） | ✅ 可能 |
| 分離性 | 同一ネットワーク上のすべてと通信可 | 同じネットワークに属するもののみ |
| Docker Compose | 自動でユーザー定義ネットワークを作成 | ✅ |

> **Docker Compose を使うと**、サービス名がそのままホスト名になります（例: `db` サービスには `db:5432` でアクセス可能）。手動でネットワーク設定を書く機会は大幅に減ります（→ Chapter 7）。

## Chapter 5: Volume（ボリューム）

### 1. コンテナのデータは消える

Chapter 3 で触れたように、コンテナの書き込み可能レイヤーはコンテナが削除されると消えます。データベースのデータやアップロードされたファイルをコンテナ内に直接書き込むと、コンテナを再起動・更新するたびにデータが失われます。

**Docker Volume** はこの問題を解決するための「コンテナの外でデータを永続化する」仕組みです。

### 2. データ保持の 3 つの方法

| 方式 | 仕組み | 向いている用途 |
|------|--------|-------------|
| **Volume** | Docker が管理する専用ストレージ領域（`/var/lib/docker/volumes/`） | DB データ・アプリ生成ファイルの永続化 |
| **Bind Mount** | ホストの任意ディレクトリをコンテナ内にマウント | 開発中のソースコードのホットリロード |
| **tmpfs Mount** | メモリ上に一時領域を作成（Linux 専用） | 秘密情報など永続化したくない一時データ |

```
ホスト OS
┌──────────────────────────────────────────────────┐
│                                                  │
│  /var/lib/docker/volumes/mydata/  ←── Volume     │
│                 ↕                                │
│  ┌──────────────────┐                            │
│  │   Container      │                            │
│  │   /app/data  ────┼── マウントポイント          │
│  └──────────────────┘                            │
│                                                  │
│  ~/projects/myapp/  ←── Bind Mount              │
│        ↕                                        │
│  ┌──────────────────┐                            │
│  │   Container      │                            │
│  │   /app/src  ─────┼── マウントポイント          │
│  └──────────────────┘                            │
└──────────────────────────────────────────────────┘
```

### 3. Volume vs Bind Mount の選び方

| 観点 | Volume | Bind Mount |
|------|--------|------------|
| 管理 | Docker が管理（ホスト構造に依存しない） | ホストのパスを直接指定 |
| ポータビリティ | ✅ 高い（どの OS でも同じ） | ❌ ホストのパスに依存 |
| パフォーマンス | ✅ Docker 最適化 | 環境によって差がある |
| バックアップ | `docker volume` コマンドで管理 | ホストのファイルと同じ |
| 推奨場面 | 本番・CI でのデータ永続化 | 開発中のソースコード共有 |

### 4. Volume のライフサイクル

Volume はコンテナと独立して存在します。コンテナを削除しても Volume は残ります。

```
docker volume create mydata    # Volume 作成
docker run -v mydata:/app/data ...  # コンテナにマウント
docker rm コンテナ              # コンテナ削除 → Volume は残る
docker volume rm mydata         # Volume を明示的に削除
```

> ⚠ **よくあるアンチパターン**: 開発で Bind Mount を使ってホストのコードをマウントするのは一般的ですが、本番環境で同じ構成を使うとホストのパス構造への依存が生まれます。本番では必ず Volume を使いましょう。

## Chapter 6: Registry（レジストリ）

### 1. イメージの「物流センター」

**Registry（レジストリ）** はビルドしたイメージを保存・配布するためのサーバーです。開発者がビルドしたイメージを push（送信）し、本番サーバーや CI がそれを pull（取得）することで、どこでも同じイメージを使えます。

```
開発者 PC                                本番サーバー / CI
  ↓ docker build                              ↓ docker pull
  ↓ docker push                               ↓ docker run
  ┌──────────────────────────────────┐
  │           Registry               │
  │  myapp:1.0  myapp:1.1  nginx:... │
  └──────────────────────────────────┘
```

### 2. Docker Hub と主要レジストリ

| レジストリ | 提供元 | 特徴 |
|-----------|--------|------|
| **Docker Hub** | Docker 社 | デフォルトレジストリ。公式イメージ（nginx・postgres・python 等）が揃う |
| **Amazon ECR** | AWS | AWS 統合・IAM 認証。ECS/EKS との親和性が高い |
| **Google Artifact Registry** | Google Cloud | GKE との統合・脆弱性スキャン内蔵 |
| **GitHub Container Registry** | GitHub | GitHub Actions との連携が自然 |
| **Azure Container Registry** | Azure | AKS・Azure DevOps との統合 |
| **self-hosted（自前）** | — | Harbor 等で社内に構築。外部流出NG の場合 |

### 3. イメージのタグ管理

レジストリはイメージを `リポジトリ名:タグ` で管理します。タグはバージョン管理に使います。

| タグ戦略 | 例 | メリット・注意点 |
|---------|---|---------------|
| セマンティックバージョン | `myapp:1.2.3` | 明確なバージョン管理。推奨 |
| コミットハッシュ | `myapp:a3f5c9d` | CI での自動タグ付けに向く |
| 環境名 | `myapp:staging` | 環境ごとのデプロイ管理に使う |
| `latest` | `myapp:latest` | 「最新」を示すが再現性がない。本番での固定運用には不向き |

> ⚠ **`latest` タグの罠**: `latest` は自動更新されます。`docker pull nginx` は実行タイミングによって異なるバージョンを取得するリスクがあります。本番では必ずバージョンタグで固定してください。

### 4. Docker Hub の公式イメージ

Docker Hub には **Official Images**（Docker 社が保守する信頼性の高いイメージ）が揃っています。一からイメージを作る前に公式イメージを確認しましょう。

| カテゴリ | 代表的な公式イメージ |
|---------|-------------------|
| OS ベース | `ubuntu`・`debian`・`alpine`・`scratch` |
| 言語ランタイム | `node`・`python`・`golang`・`java` |
| Web サーバー | `nginx`・`httpd` |
| データベース | `postgres`・`mysql`・`redis`・`mongodb` |

## Chapter 7: Docker Compose

### 1. 「複数コンテナをまとめて管理する」問題

実用的なアプリはほぼ必ず複数のコンテナで構成されます。Web サーバー・アプリケーションサーバー・データベース・キャッシュサーバー……これらを個別に `docker run` で起動するのは煩雑で、依存関係の管理も大変です。

**Docker Compose** はこの問題を `compose.yaml` という単一ファイルで解決します。

```
compose.yaml で定義        docker compose up
       ↓
  ┌────────────────────────────────────────┐
  │  Application                           │
  │  ┌──────────┐  ┌──────┐  ┌─────────┐  │
  │  │  nginx   │  │  app │  │   db    │  │
  │  │  :80     │  │ :3000│  │ :5432   │  │
  │  └──────────┘  └──────┘  └─────────┘  │
  │       ↑自動でネットワーク・依存関係設定  │
  └────────────────────────────────────────┘
```

### 2. compose.yaml の基本構造

```yaml
services:            # ← コンテナを「サービス」として定義
  web:
    image: nginx:1.25
    ports:
      - "80:80"      # ← ホスト:コンテナ のポートマッピング
    depends_on:
      - app          # ← app が起動してから web を起動

  app:
    build: .         # ← ローカルの Dockerfile からビルド
    environment:
      DATABASE_URL: postgres://db:5432/mydb
    volumes:
      - .:/app       # ← Bind Mount（開発時のホットリロード）

  db:
    image: postgres:16
    volumes:
      - pgdata:/var/lib/postgresql/data  # ← Volume でデータ永続化

volumes:
  pgdata:            # ← Volume の宣言
```

### 3. Docker Compose の主要コマンド

| コマンド | 動作 |
|---------|------|
| `docker compose up` | サービスを全起動（`-d` でバックグラウンド） |
| `docker compose down` | サービスを停止・コンテナ削除（Volume は残る） |
| `docker compose down -v` | Volume も含めて全削除 |
| `docker compose logs -f` | ログをリアルタイムで表示 |
| `docker compose exec app bash` | 実行中コンテナにシェルで入る |
| `docker compose ps` | サービスの状態を確認 |

### 4. Compose の位置づけ

| ツール | 用途 | 規模 |
|--------|------|------|
| `docker run` | 単一コンテナのテスト・一時起動 | 1 コンテナ |
| **Docker Compose** | 開発環境・単一ホストの複数コンテナ管理 | 数〜十数コンテナ |
| Kubernetes | 本番の大規模オーケストレーション | 数十〜数千コンテナ |

## Chapter 8: エコシステムと次の一歩

### 1. Docker とその周辺技術の全体図

Docker を起点に広がるエコシステムを俯瞰します。

```
                 ┌──────────────────┐
                 │   開発者ツール    │
                 │  Docker Desktop  │
                 │  VS Code DevContainer│
                 └────────┬─────────┘
                          │
              ┌───────────▼──────────────┐
              │    Docker Engine         │
              │  （build / run / push）   │
              └───┬───────────┬──────────┘
                  │           │
        ┌─────────▼──┐   ┌────▼──────────────┐
        │  Registry  │   │  Orchestration     │
        │ Docker Hub │   │  Kubernetes        │
        │ ECR / GCR  │   │  Docker Swarm      │
        └────────────┘   └────────────────────┘
```

### 2. Kubernetes との関係

Docker と Kubernetes は競合ではなく**補完関係**です。

| 役割 | ツール |
|------|--------|
| **イメージのビルド・配布** | Docker（Build + Registry） |
| **単一ホストの実行** | Docker Engine / Docker Compose |
| **複数ホストのオーケストレーション** | Kubernetes |

Kubernetes はかつて Docker Engine をランタイムとして直接利用していましたが、v1.24 以降は `containerd` や `CRI-O` などの OCI 準拠ランタイムを直接使います。ただし Docker でビルドした **OCI 準拠イメージ**は Kubernetes 上でそのまま動作します。

「Docker を使えば Kubernetes も学べる」ではなく、「Docker でビルド・管理し、Kubernetes でスケールさせる」という役割分担を覚えておきましょう。

### 3. 競合・代替ツールとの比較

| ツール | 特徴 | 向いている場面 |
|--------|------|-------------|
| **Docker** | 最も広く普及・豊富なエコシステム | デファクトスタンダードとして |
| **Podman** | デーモンレス・デフォルト rootless でセキュア | セキュリティ要件が厳しい環境 |
| **containerd** | Kubernetes のデファクトランタイム | Kubernetes の内部ランタイムとして |
| **Buildah** | デーモンレスイメージビルド | CI でのイメージビルド特化 |

### 4. よくある誤解・アンチパターンまとめ

| アンチパターン | 何が問題か | 対策 |
|-------------|-----------|------|
| `FROM ubuntu:latest` | ビルドごとに異なるベースになりうる | バージョン固定（`ubuntu:22.04`） |
| コンテナに秘密情報を ENV で埋め込む | イメージ履歴に残る | Docker Secrets / 環境変数ファイルで注入 |
| 1 コンテナに複数プロセスを詰め込む | スケール・ログ管理が困難 | 1 コンテナ 1 プロセス原則を守る |
| 本番で Bind Mount を使う | ホストパスへの依存・移植性低下 | Volume を使う |
| コンテナ内でデータを直接変更 | 再デプロイで消える | イミュータブル運用・Volume 分離 |
| 巨大なイメージをそのままビルド | 転送・起動が遅い | マルチステージビルドで最終イメージを軽量化 |
