# GitHub Projects チュートリアル

> Issue や PR は使っているが Projects はまだ触ったことがない方向けのハンズオン教材。
> Chapter を順に進めれば、プロジェクト作成から自動化・チーム運用まで一通り身につきます。
> より深く知りたくなったら → `github-projects-best-practices.md`（準備中）

---

## Chapter 0: GitHub Projects とは

### 1. Issue/PR だけでは何が足りないか

GitHub で開発していると、Issue でタスクを管理し、PR でコードをレビューする流れは自然に身につきます。しかし少し規模が大きくなると、こんな問題が出てきます。

```
「今スプリントで何が進んでいて、何が止まっているか一目でわからない」
「どの Issue が誰の手元にあるか確認するのに検索が必要」
「締め切りが近いものをひとまとめに見たい」
```

Issue の一覧ページはリポジトリごとに分断されており、横断的な進捗管理には向きません。ここに **GitHub Projects** が入ってきます。

### 2. GitHub Projects とは

公式の定義を借りると：

> "an adaptable table, board, and roadmap that integrates with your issues and pull requests on GitHub"

一言でいえば「**Issue/PR と双方向に連動する適応型スプレッドシート**」です。

```
Issue/PR ────────────────────────────
              ↕ 双方向で同期
GitHub Projects ─── Table ビュー
                 ─── Board ビュー（カンバン）
                 ─── Roadmap ビュー（タイムライン）
```

プロジェクト上でアサイニーやステータスを変更すると Issue 側にも即時反映されます。逆も同様です。二重管理は不要です。

### 3. 旧 Projects（Classic）との違い

GitHub Projects には世代があります。現在の Projects（通称 **Projects v2**）は 2022 年に正式リリースされた新世代です。

| | Projects Classic（旧） | Projects（現在） |
|---|---|---|
| フィールドのカスタマイズ | なし | 最大 50 フィールド追加可能 |
| ビューの種類 | カンバンのみ | Table / Board / Roadmap |
| 自動化 | 限定的 | Built-in Workflow + GitHub Actions 連携 |
| 複数リポジトリ横断 | 難しい | ネイティブ対応 |
| Insights（グラフ） | なし | あり |

本教材では現在の Projects（v2）を扱います。

### 4. 3 つの主要概念

Projects を使う上で最初に押さえておきたい概念が 3 つあります。

| 概念 | 説明 |
|------|------|
| **Item（アイテム）** | プロジェクトに追加する作業の単位。Issue・PR・Draft Issue の 3 種類 |
| **Field（フィールド）** | アイテムに付与するメタデータ列。Status・Priority・担当者など |
| **View（ビュー）** | アイテムの見せ方。Table・Board・Roadmap の 3 形式。1 プロジェクトに複数作れる |

```
プロジェクト
├── Item（Issue, PR, Draft Issue）
│   ├── Field: Status = "In Progress"
│   ├── Field: Priority = "High"
│   └── Field: Assignee = @tkd
└── View
    ├── Table ビュー（全フィールドを一覧）
    ├── Board ビュー（ステータスでカラム分け）
    └── Roadmap ビュー（期日でタイムライン表示）
```

### 5. どんな場面で使うか

Projects はチームの規模や開発スタイルを問わず使えますが、特に効果的な場面があります。

| 場面 | 使い方 |
|------|--------|
| スプリント管理 | Board ビュー + Iteration フィールドで 1 週間単位の作業を管理 |
| リリース計画 | Roadmap ビューで締め切りと進捗をタイムラインで可視化 |
| バグトラッキング | Table ビューで Priority フィールドを使って優先度別に並べる |
| 個人タスク管理 | Draft Issue で思いついたタスクをすぐ追加して後で整理 |

> **旧 Projects Classic を使っていた方へ**: メニューの場所やビューの構成が大きく変わっています。最初は「別ツールを覚える」くらいの気持ちで臨むとスムーズです。

### 6. よくあるアンチパターン（Chapter 0 版）

- **Issue と Projects を二重管理する**: 連動しているので Projects 上で編集すれば Issue も更新されます。両方を別々に更新する必要はありません。
- **最初から完璧な設計をしようとする**: フィールドは後から追加・削除できます。まず Status だけで動かし始めるのが得策です。

## Chapter 1: プロジェクトを作って 5 分で成功体験

### 1. プロジェクトの作り方

Projects は **個人アカウント** と **Organization（組織）** のどちらにも作成できます。

#### 個人アカウントの場合

1. GitHub 右上のアイコン → **Your profile**
2. **Projects** タブ → **New project**
3. テンプレートを選んで名前を入力 → **Create project**

#### Organization の場合

1. Organization のトップページ → **Projects** タブ → **New project**
2. 以降は個人と同じ手順

> **どちらを使うべきか**: チームで使うなら Organization プロジェクト一択です。個人プロジェクトにはチームメンバーを招待できません。

---

### 2. テンプレートの選び方

新規作成時にテンプレートを選べます。迷ったら **Board** か **Table** から始めるのがおすすめです。

| テンプレート | 向いている使い方 |
|-------------|-----------------|
| **Table** | フィールドを列として並べ、スプレッドシート感覚で管理したい |
| **Board** | カンバン形式で「Todo / In Progress / Done」を視覚的に管理したい |
| **Roadmap** | マイルストーンや締め切りをタイムラインで把握したい |

後からビューは何でも追加できるので、テンプレート選びに時間をかけすぎなくて大丈夫です。

---

### 3. 5 分で成功体験：プロジェクトを動かしてみる

実際に手を動かしましょう。以下の手順で「Issue を追加して Status を変更する」という一連の操作を体験します。

#### ステップ 1: プロジェクトを作成する

```
1. GitHub の自分のプロフィールページを開く
2. Projects タブ → New project をクリック
3. テンプレートで "Board" を選択
4. 名前に "my-first-project" と入力
5. Create project をクリック
```

Board テンプレートを選ぶと、最初から **Todo / In Progress / Done** の 3 カラムが用意された状態で開きます。

#### ステップ 2: アイテムを追加する

アイテムの追加方法は複数あります。まずは一番簡単な方法を試してください。

**方法 A: URL を貼り付ける（既存の Issue がある場合）**

```
1. ボード下部の "+ Add item" をクリック
2. Issue または PR の URL を貼り付けて Enter
```

**方法 B: Draft Issue を作る（手軽に試したい場合）**

```
1. "Todo" カラム下部の "+ Add item" をクリック
2. タイトルを入力して Enter（例: "最初のタスク"）
```

Draft Issue はリポジトリに紐付かない仮のアイテムです。「とりあえず書き出す」場所として使えます。

#### ステップ 3: Status を動かす

アイテムを追加できたら、ドラッグ＆ドロップで **In Progress** カラムへ移動させてみてください。

```
Todo の "最初のタスク" → In Progress カラムへドラッグ
```

Status が変わりました。これが Projects の基本操作です。

Table ビューで確認したい場合は、画面上部の **+ New view** → **Table** を選ぶと、同じアイテムが表形式で並んでいるのを確認できます。

---

### 4. アイテムを追加する 5 つの方法

実務では用途によって追加方法を使い分けます。

| 方法 | 手順 | 向いている場面 |
|------|------|---------------|
| **URL 貼り付け** | `+ Add item` → URL ペースト | 既存の Issue/PR を 1 件ずつ追加 |
| **`#` 検索** | `+ Add item` → `#` を入力して検索 | リポジトリ内の Issue を名前で探して追加 |
| **リポジトリから一括追加** | `+ Add item` → "Add item from repository" | フィルタしながら複数件まとめて追加 |
| **Issue/PR のサイドバー** | Issue を開く → 右サイドバーの Projects | 個別 Issue 側からプロジェクトに割り当て |
| **Draft Issue** | カラム下部に直接入力 | まだ Issue 化していないアイデアをメモ |

> **Draft Issue の注意点**: Draft Issue はリポジトリ・ラベル・マイルストーンを持てません。チームへの通知も発生しません。本格的に取り組む前に Issue に変換しましょう（アイテム右クリック → "Convert to issue"）。

---

### 5. よくあるアンチパターン

- **Draft Issue を大量に作って放置する**: 後で整理しようと思っているうちに溜まっていきます。追加したらその週中に Issue 化か削除かを判断する習慣をつけると管理しやすくなります。
- **Organization で作るべきところを個人プロジェクトで作ってしまう**: 後からメンバーを招待できないため、作り直しが必要になります。

---

### 練習課題

1. **基本**: Board テンプレートでプロジェクトを 1 つ作成し、Draft Issue を 3 件追加して、それぞれ Todo / In Progress / Done に振り分けてください。
2. **応用**: 既存のリポジトリから Issue または PR を 2 件以上追加してください（URL 貼り付けと `#` 検索の 2 通りで試してください）。
3. **確認**: Table ビューを追加して、同じアイテムが Board と Table 両方で見えることを確認してください。

## Chapter 2: アイテムの追加と管理

### 1. アイテムの 3 種類

プロジェクトに追加できるアイテムは 3 種類あります。

| 種類 | 説明 | 向いている場面 |
|------|------|---------------|
| **Issue** | リポジトリに紐付いた作業単位。ラベル・担当者・マイルストーンを持てる | 実際の開発タスク・バグ・機能要望 |
| **Pull Request** | コードの変更に紐付いたアイテム。マージ状態も Status に反映できる | コードレビューの進捗管理 |
| **Draft Issue** | リポジトリ非紐付けの仮アイテム。通知なし | アイデアのメモ書き・後で整理するタスク |

```
Draft Issue ──変換──▶ Issue（リポジトリを選択）
                        ↑
              ラベル・担当者・マイルストーンが使えるようになる
```

---

### 2. Draft Issue を Issue に変換する

Draft Issue は「とりあえず書き出す」場所ですが、チームで動かすには Issue に変換が必要です。

#### Table ビューでの変換

```
1. Draft Issue の行にある ◀ アイコンをクリック
2. "Convert to issue" を選択
3. 変換先のリポジトリを選択 → Enter
```

#### Board ビューでの変換

```
1. カードの … メニューをクリック
2. "Convert to issue" を選択
3. 変換先のリポジトリを選択 → Enter
```

変換後は通常の Issue として扱われ、ラベル・マイルストーン・担当者の設定、チームへの通知が有効になります。

> **変換先リポジトリを間違えると**: Issue 自体は削除できますが、作業履歴が残ります。変換前に必ず正しいリポジトリを確認してください。

---

### 3. アーカイブと削除の違い

完了したアイテムを片付けるとき、**アーカイブ** と **削除** の 2 つの選択肢があります。

| 操作 | データの扱い | ビューへの表示 | 元に戻せるか |
|------|-------------|--------------|-------------|
| **アーカイブ** | プロジェクト内に残る | 非表示になる | 復元できる |
| **削除（Remove from project）** | プロジェクトから消える | なし | 戻せない |

**操作手順（共通）**:

```
Table ビュー: 行を右クリック → "Archive" または "Remove from project"
Board ビュー: カードの … メニュー → "Archive" または "Remove from project"
複数選択:    Ctrl/Cmd + クリックで選択 → 右クリックメニュー
```

> **アーカイブを使う場面**: 「完了したが履歴として残しておきたい」スプリントの Done アイテムなど。
> **削除を使う場面**: 「誤って追加した」「重複した」など、最初から存在しなかった扱いにしたい場合。

> **Remove from project の注意点**: この操作はプロジェクトとの紐付けを外すだけです。元のリポジトリの Issue や PR は削除されず、そのまま残ります。Issue 自体を閉じたい場合は、別途 Issue 画面から "Close issue" を行ってください。

⚠ **注意**: アーカイブしたアイテムも **上限 50,000 件** にカウントされます。長期プロジェクトでは定期的に削除も組み合わせましょう。

---

### 4. アーカイブ済みアイテムを復元する

```
1. プロジェクトの … メニュー → "Archived items"
2. 復元したいアイテムを選択
3. "Restore" をクリック
```

アーカイブ一覧はフィルタも使えるので、タイトルで絞り込んで探せます。

---

### 5. コマンドパレットで素早く操作する

`Cmd + K`（Mac）/ `Ctrl + K`（Windows）でコマンドパレットが開きます。アイテム追加・フィールド変更・ビュー切り替えをキーボードだけで操作できます。慣れてくると手放せなくなります。

---

### 6. よくあるアンチパターン

- **Done になった Issue をいつまでもビューに残す**: Board が Done だらけになると「今何をすべきか」が見えにくくなります。スプリントが終わったら Done カラムを一括アーカイブする習慣をつけましょう。
- **削除と勘違いしてアーカイブし続ける**: アーカイブ済みアイテムも上限にカウントされます。「もう見ない」アイテムは削除しても問題ありません。

---

### 練習課題

1. **基本**: Chapter 1 で作ったプロジェクトに Draft Issue を 2 件追加し、1 件を Issue に変換してください（変換先リポジトリは自分のリポジトリを使用）。
2. **応用**: In Progress か Done にあるアイテムを 1 件アーカイブし、その後 "Archived items" から復元してください。
3. **確認**: Table ビューで複数のアイテムを `Cmd/Ctrl + クリック` で選択し、まとめてアーカイブしてください。

## Chapter 3: フィールドでアイテムを整理する

### 1. フィールドとは

フィールドはアイテムに付与できる「列」です。スプレッドシートの列と同じ感覚で、好きな情報をアイテムに紐付けられます。

フィールドには **組み込みフィールド** と **カスタムフィールド** の 2 種類があります。

| 種類 | 例 | 変更・削除 |
|------|-----|----------|
| **組み込み（Built-in）** | Title / Assignees / Status / Labels / Milestone | 削除不可・Issue と自動同期 |
| **カスタム（追加）** | Priority / Story Points / 締め切り日 / Sprint | 追加・名前変更・削除が自由 |

---

### 2. カスタムフィールドの種類と使いどころ

| 型 | 向いている用途 | 例 |
|----|--------------|-----|
| **Text** | 自由記述のメモ・補足 | 対応メモ、外部チケット番号 |
| **Number** | 数値管理 | ストーリーポイント、工数（時間） |
| **Date** | 日付指定 | 締め切り日、リリース予定日 |
| **Single Select** | 固定の選択肢から 1 つ選ぶ | Priority（High / Medium / Low） |
| **Iteration** | スプリントなどの繰り返し期間 | Sprint 1 / Sprint 2 / Sprint 3 |

> **最初はフィールドを絞る**: 最初から 10 個フィールドを作ると管理が煩雑になります。まず **Status**（組み込み）と **Priority**（Single Select）の 2 つだけで運用を始め、必要に応じて追加していくのが得策です。

---

### 3. フィールドを追加する

```
1. Table ビューを開く
2. 列の一番右にある「+」をクリック
3. "New field" を選択
4. フィールド名を入力し、型を選択
5. Save をクリック
```

---

### 4. Single Select フィールドを作る（Priority の例）

Priority フィールドを Single Select で作る手順です。Board ビューのカラム分けにも使える最重要フィールドです。

```
1. 「+」→ "New field" → 名前に "Priority" と入力
2. 型で "Single select" を選択
3. オプションを追加:
   - "🔴 High"
   - "🟡 Medium"
   - "🟢 Low"
4. 各オプションに色を設定（任意）
5. Save をクリック
```

作成後はアイテムの Priority セルをクリックするだけで選択肢が出てきます。

**色の設定**:

```
プロジェクト設定（右上 ⚙）→ Fields → Priority → 各選択肢の Color をクリック
```

---

### 5. Iteration フィールドを作る（Sprint の例）

スプリント管理をしたい場合は Iteration フィールドが便利です。

```
1. 「+」→ "New field" → 名前に "Sprint" と入力
2. 型で "Iteration" を選択
3. 開始日と期間（例: 2週間）を設定
4. Save をクリック → Sprint 1 / Sprint 2 / Sprint 3 が自動生成される
```

**Iteration フィールドの便利な機能**:

- スプリント間に「休暇期間」を挿入できる（"Insert break"）
- Table ビューで "Group by Sprint" にすると、スプリントごとにアイテムがまとめて表示される
- 複数のアイテムをまとめて別スプリントに移動できる

```
Table ビューで Group by: Sprint にする手順:
1. ビュー上部の ▼ → "Group by"
2. "Sprint" を選択
```

---

### 6. フィールドを編集・削除する

```
プロジェクト設定（右上 ⚙）→ "Fields" タブ
→ 対象フィールドをクリックして編集 / "Delete field" で削除
```

⚠ **組み込みフィールド（Title / Assignees / Status など）は削除できません。** カスタムフィールドのみ削除可能です。

---

### 7. よくあるアンチパターン

- **フィールドを作りすぎる**: 使われないフィールドが増えると Table ビューが横に広がりすぎて見づらくなります。「このフィールドが埋まっていないと困る場面はあるか？」を問いかけてから追加しましょう。
- **Single Select の選択肢が多すぎる**: Priority に 5 段階以上作ると判断コストが高くなります。High / Medium / Low の 3 段階で十分なことがほとんどです。
- **Iteration を作ったがスプリントレビューをしない**: フィールドを作っても振り返りのサイクルがなければ形骸化します。Chapter 5 の自動化と組み合わせると効果的です。

---

### 練習課題

1. **基本**: "Priority" という名前の Single Select フィールドを作成し、High / Medium / Low の 3 つの選択肢（色付き）を追加してください。プロジェクトのアイテムすべてに Priority を設定してください。
2. **応用**: "Sprint" という名前の Iteration フィールドを 2 週間サイクルで作成し、Table ビューで "Group by Sprint" にして表示してください。
3. **確認**: Table ビューで Priority の列ヘッダーをクリックしてソートし、High のアイテムが上に来ることを確認してください。

## Chapter 4: ビューで見方を変える

### 1. 3 つのビューの特徴

1 つのプロジェクトに **複数のビュー** を追加できます。同じアイテムを場面によって切り替えて表示するのが Projects の強みです。

| ビュー | 特徴 | 向いている場面 |
|--------|------|---------------|
| **Table** | スプレッドシート形式。全フィールドを列として並べる | 一覧確認・フィールドの一括編集・ソート |
| **Board** | カンバン形式。列間のドラッグで Status が自動更新 | 進捗の視覚的な把握・デイリースクラム |
| **Roadmap** | タイムライン形式。日付で期間を可視化 | リリース計画・スプリント俯瞰 |

```
同じアイテム群を…
  Table → 全フィールドを一覧で確認したいとき
  Board → 「誰が何をやっているか」を朝会で確認するとき
  Roadmap → 「いつ何が終わるか」をマネージャーに見せるとき
```

---

### 2. ビューを追加・切り替える

```
新規追加: 画面上部のタブ右端「+ New view」をクリック
切り替え: タブをクリック
レイアウト変更: View options（▼）→ Layout → 種類を選択
並び替え: タブをドラッグ&ドロップ
複製: View options → "Duplicate view"
削除: View options → "Delete view"
```

変更を加えるとビュー名の横に **●（ドット）** が付きます。保存するには View options → **"Save changes"** をクリックしてください。保存しないまま別ビューに移動すると変更が破棄されます。

---

### 3. Filter（フィルタ）でアイテムを絞り込む

ビュー上部の検索バーにフィルタ条件を入力します。

**基本構文**:

```
field:value
```

**よく使うフィルタ例**:

```bash
assignee:@me                    # 自分にアサインされたアイテム
status:"In Progress"            # ステータスが In Progress
label:bug                       # bug ラベルのついたアイテム
priority:High                   # Priority が High
-assignee:octocat               # octocat 以外のアイテム（否定）
no:assignee                     # 担当者未設定のアイテム
is:open                         # オープンな Issue/PR のみ
assignee:@me status:"In Progress"  # AND（スペース区切り）
```

> **`@me` と `@today`**: ログイン中のユーザー自身・今日の日付を指す特殊キーワードです。チームで同じビューを共有しても、各自が自分のタスクだけを表示できます。

---

### 4. Group by でまとめる

フィールドの値ごとにアイテムをグループ分けできます。

```
View options（▼）→ "Group by" → フィールドを選択
```

**Group by の活用例**:

| Group by の対象 | 使い道 |
|----------------|--------|
| **Status** | Table ビューをカンバン的に見る（Board の代替） |
| **Assignee** | 誰がどれだけ抱えているかを確認（負荷分散） |
| **Priority** | 優先度別に作業を整理 |
| **Sprint（Iteration）** | スプリントごとの作業量を把握 |

グループ化した状態でアイテムを別グループへドラッグすると、そのグループのフィールド値が自動で設定されます。

---

### 5. Sort（ソート）で並び替える

```
View options（▼）→ "Sort by" → フィールドを選択 → 昇順 / 降順
```

複数フィールドによる **多段ソート** も可能です。たとえば「Priority の高い順、同じ Priority なら締め切り日が近い順」といった並び替えができます。

---

### 6. Roadmap ビューの設定

Roadmap ビューを使うには、**日付を持つフィールド**（Date または Iteration）が必要です。

```
1. ビューのレイアウトを "Roadmap" に変更
2. カレンダーアイコン → "Date fields" をクリック
3. 開始日・終了日に使うフィールドを指定
   （例: 開始日 = "Sprint 開始", 終了日 = "締め切り日"）
4. ズームレベルを Month / Quarter / Year から選択
```

**Roadmap 固有の便利機能**:

- **スライス（Slice by）**: フィールド値ごとにタイムラインをパネル分割して表示
- **マーカー**: イテレーションやマイルストーンの区切り線をタイムライン上に表示

---

### 7. ビューの設計パターン

実務でよく使われるビュー構成の例を紹介します。

```
ビュー 1: Board（デイリー用）
  → Status でカラム分け、担当者フィルタなし

ビュー 2: 自分のタスク（個人用）
  → Table + Filter: assignee:@me is:open + Sort: Priority 高い順

ビュー 3: Sprint 計画（週次レビュー用）
  → Table + Group by: Sprint + Sort: Priority

ビュー 4: Roadmap（マネージャー向け）
  → Roadmap + ズーム: Quarter
```

---

### 8. よくあるアンチパターン

- **ビューを作りすぎる**: ビューが 10 個を超えると管理しきれなくなります。「誰が・何の目的で使うビューか」を決めてから作りましょう。
- **フィルタを保存し忘れる**: 変更後に "Save changes" を押さないと次回開いたときに元に戻ります。ビュー名横の ● を見逃さないようにしてください。
- **Roadmap に日付フィールドを設定しないまま使おうとする**: アイテムがタイムラインに表示されません。Date または Iteration フィールドを先に作っておく必要があります。

---

### 練習課題

1. **基本**: 既存のビューに `assignee:@me` フィルタを追加して保存し、自分にアサインされたアイテムだけが表示されることを確認してください。
2. **応用**: Table ビューを「Priority 高い順・同 Priority なら締め切り日が近い順」の多段ソートで設定してください。
3. **複合**: 以下の 3 つのビューを作成してください。
   - Board ビュー（Status でカラム分け）
   - Table ビュー（Group by: Priority）
   - Roadmap ビュー（Sprint Iteration を日付フィールドに指定）

## Chapter 5: Built-in Workflow で自動化する

### 1. 自動化が必要な理由

手動で Status を更新し続けると、こんな問題が起きます。

```
「Issue をクローズしたのに Board の Done カラムに移動し忘れた」
「PR がマージされたが、プロジェクトのステータスが In Progress のまま」
「完了タスクが Board に溜まって今やるべきことが見えなくなった」
```

GitHub Projects には **Built-in Workflow**（組み込み自動化）が用意されています。設定はトグルを ON にするだけです。

---

### 2. Built-in Workflow の種類

| ワークフロー名 | トリガー | 自動で行われること |
|--------------|---------|-----------------|
| **Item closed** | Issue/PR がクローズされた | Status を "Done" に変更 |
| **Pull request merged** | PR がマージされた | Status を "Done" に変更 |
| **Auto-add to project** | Issue/PR が作成・更新された | 条件一致したら自動でプロジェクトに追加 |
| **Auto-archive items** | 条件一致アイテムを定期チェック | 対象アイテムを自動アーカイブ |

デフォルトで有効になっているのは **"Item closed"** と **"Pull request merged"** の 2 つです。残りは手動で有効化します。

---

### 3. Workflow を有効化・無効化する

```
1. プロジェクト右上の「…」メニュー → "Workflows" をクリック
2. 有効化したいワークフローを選択
3. "Edit" をクリック
4. 設定を確認して "Save and turn on workflow" をクリック
```

無効化したい場合は、同じ画面のトグルを OFF にするだけです。

---

### 4. Auto-add to project：条件に合う Issue/PR を自動追加する

リポジトリに Issue が作られるたびに手動でプロジェクトに追加するのは手間です。**Auto-add to project** を使えば、条件を満たす Issue/PR を自動でプロジェクトに追加できます。

**設定手順**:

```
1. Workflows → "Auto-add to project" → "Edit"
2. 対象リポジトリを選択
3. フィルター条件を入力（例: is:issue label:bug）
4. "Save and turn on workflow"
```

**使えるフィルター条件**:

```bash
is:issue                  # Issue のみ対象
is:pr                     # PR のみ対象
label:bug                 # bug ラベルが付いているもの
label:bug,enhancement     # bug または enhancement ラベル（OR）
-label:wontfix            # wontfix ラベルが付いていないもの
assignee:username         # 特定ユーザーにアサインされたもの
no:assignee               # 担当者未設定のもの
is:open                   # オープンなもののみ
```

> **Auto-add の上限（プランによって異なります）**:
> Free: 1 件 / Pro・Team: 5 件 / Enterprise: 20 件

---

### 5. Auto-archive items：古いアイテムを自動で片付ける

Done のアイテムが溜まってきたら、Auto-archive で自動的にアーカイブできます。

**設定手順**:

```
1. Workflows → "Auto-archive items" → "Edit"
2. アーカイブ条件を選択:
   - is:closed（クローズ済み）
   - reason:completed または reason:not planned
   - 最終更新から 14 日 / 3 週間 / 1 ヶ月 経過
3. "Save and turn on workflow"
```

⚠ **注意**: Auto-archive を有効化すると、既存の条件一致アイテムも対象になります。大量のアイテムがある場合は処理に時間がかかる場合があります。

---

### 6. どこまで自動化すべきか

自動化は便利ですが、やりすぎると「気づいたらアイテムが消えていた」という混乱を招きます。

| おすすめする自動化 | 理由 |
|-----------------|------|
| Item closed → Done | 手動更新の最大の忘れポイント |
| PR merged → Done | マージ後の更新は必ず漏れる |
| Auto-archive（2 週間以上更新なし） | Board の視認性を保てる |

| 慎重にすべき自動化 | 理由 |
|-----------------|------|
| Auto-add（全 Issue を追加） | プロジェクトに無関係な Issue まで入ってくる場合がある |
| Auto-archive（短すぎる期間） | まだ作業中のアイテムが消えることがある |

---

### 7. GitHub Actions 連携（発展）

Built-in Workflow でカバーできない複雑なルールは **GitHub Actions** で実装できます。

```yaml
# .github/workflows/add-to-project.yml の例
name: Add to project
on:
  issues:
    types: [opened]
jobs:
  add-to-project:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/add-to-project@v1
        with:
          project-url: https://github.com/orgs/YOUR_ORG/projects/1
          github-token: ${{ secrets.ADD_TO_PROJECT_PAT }}
```

⚠ **認証トークンの注意**: `GITHUB_TOKEN` はプロジェクトへのアクセス権がありません。Organization プロジェクトには **GitHub App**、個人プロジェクトには **Personal Access Token**（`project` スコープ付き）が必要です。

---

### 8. よくあるアンチパターン

- **全部の Workflow をとりあえず ON にする**: 動作を理解せずに有効化すると、意図しないタイミングでアイテムが移動・アーカイブされます。1 つずつ動作確認しながら有効化しましょう。
- **Auto-add の条件を広くしすぎる**: `is:issue` だけを条件にすると、全リポジトリの全 Issue がプロジェクトに入ってきます。ラベルやリポジトリで絞り込むことを推奨します。

---

### 練習課題

1. **基本**: "Item closed" と "Pull request merged" の Workflow が有効になっていることを確認してください。Issue を 1 件クローズして、Status が自動的に "Done" に変わることを確認してください。
2. **応用**: 特定のラベル（例: `tutorial`）が付いた Issue を自動でプロジェクトに追加する Auto-add Workflow を設定してください。
3. **確認**: Auto-archive を「クローズ済みかつ 14 日以上更新なし」の条件で設定し、Workflows 一覧で有効化されていることを確認してください。

## Chapter 6: チームで使う

### 1. Organization プロジェクト vs 個人プロジェクト（再確認）

Chapter 1 で触れましたが、チームで使う場合は **Organization プロジェクト**一択です。

| | 個人プロジェクト | Organization プロジェクト |
|---|---|---|
| メンバー招待 | できない | できる |
| チーム単位の権限付与 | できない | できる |
| 複数リポジトリ横断 | 自分のリポジトリのみ | Organization 配下すべて |
| Insights（グラフ） | あり | あり |

---

### 2. 権限レベルの種類

プロジェクトへのアクセス権は 3 段階です。

| ロール | できること |
|--------|-----------|
| **Admin** | 設定変更・メンバー管理を含むすべての操作 |
| **Write** | アイテムの追加・編集・ステータス更新など |
| **Read** | 閲覧のみ（編集不可） |

**誰に何を付与するか の目安**:

```
プロジェクトオーナー・スクラムマスター → Admin
開発メンバー・デザイナー・QA          → Write
ステークホルダー・マネージャー         → Read
```

---

### 3. メンバーを招待する

```
1. プロジェクト右上「…」→ "Settings"
2. 左メニュー "Manage access" をクリック
3. "Invite collaborators" に名前・ユーザー名を入力
4. ロール（Read / Write / Admin）を選択 → "Invite" をクリック
```

> **招待できる対象**: Organization のメンバー、またはいずれかのリポジトリの Outside Collaborator である必要があります。Organization 外の人は先にリポジトリコラボレーターとして追加してください。

---

### 4. チーム単位でアクセスを付与する（推奨）

メンバーを 1 人ずつ招待するのは手間がかかります。**チーム単位**でアクセスを付与すると、チームメンバーの増減が自動で反映されるため管理コストが大幅に下がります。

```
1. "Manage access" → "Invite collaborators"
2. チーム名を入力（例: @your-org/frontend-team）
3. ロールを選択 → "Invite"
```

チームにアクセスを付与すると、そのチームの Projects ページに自動でプロジェクトが表示されます。

---

### 5. プロジェクトの可視性を設定する

```
プロジェクト "Settings" → "Manage visibility"
```

| 可視性 | 見える範囲 |
|--------|-----------|
| **Private** | 明示的に招待されたメンバーのみ |
| **Public** | Organization 全体（または外部公開） |

社内の開発プロジェクトは通常 **Private** にしておき、必要なメンバーだけを招待するのが安全です。OSS プロジェクトや公開ロードマップは **Public** にすることで外部からも参照できます。

---

### 6. Insights でプロジェクトの進捗を可視化する

**Insights** はプロジェクトのデータをグラフで可視化する機能です。

```
プロジェクト上部の "Insights" タブをクリック
```

#### Current Charts（現状スナップショット）

今の状態をグラフで見ます。

```
"New chart" → グラフタイプ・集計フィールド・フィルターを設定 → "Save"
```

**例: Status 別のアイテム数（円グラフ）**

```
グラフタイプ: Bar または Pie
X軸: Status
グループ: なし
```

#### Historical Charts（時系列推移）

過去から今日までの変化をグラフで見ます。デフォルトで **Burn up チャート** が用意されています。

```
Burn up チャートの見方:
  緑の線 = 完了アイテム数（Completed）
  赤の線 = 対象外アイテム数（Not planned）
  全体の高さ = 全アイテム数
  → 緑が全体に近づいていれば順調
```

> ⚠ **アーカイブ・削除済みアイテムはグラフに反映されません。** 完了アイテムをすぐ削除するとバーンアップが正確に出ません。Done になったアイテムは「しばらくアーカイブしておく → 期が終わったら削除」の運用が適切です。

---

### 7. チーム運用のベストプラクティス

| プラクティス | 理由 |
|------------|------|
| 個別招待よりチーム単位でアクセス付与 | メンバー増減の都度設定変更が不要になる |
| ビューを役割別に用意する | 開発者用 Board・マネージャー用 Roadmap を分ける |
| スプリントレビューで Insights を使う | Burn up で完了数を可視化し振り返りの材料にする |
| Auto-add + ラベル運用を統一する | ラベルがバラバラだと Auto-add の条件設定が複雑になる |

---

### 8. よくあるアンチパターン

- **全員を Admin にする**: 誤操作でプロジェクト設定やフィールドを壊してしまうリスクがあります。管理者以外は Write 止まりにしましょう。
- **Insights を設定したまま放置する**: グラフを作っても定期的に見る場を設けなければ形骸化します。週次レビューや Sprint 振り返りに組み込んで初めて価値が出ます。
- **チームを招待せず個人を 1 人ずつ追加する**: メンバーが増えるたびに設定変更が必要になります。最初からチーム単位で管理しましょう。

---

### 練習課題

1. **基本**: Organization プロジェクトを作成し、チームメンバー（または別のアカウント）を Write 権限で招待してください。
2. **応用**: Insights で「Status 別のアイテム数」を棒グラフで表示するチャートを作成してください。
3. **確認**: Burn up チャートを開いて、完了アイテムと未完了アイテムの数が正しく表示されていることを確認してください。

### 2. 次に学ぶべきこと

#### GitHub Actions との深い連携

Built-in Workflow では対応できない複雑なルールを自動化したい場合は、GitHub Actions と GraphQL API の組み合わせが必要です。

- `actions/add-to-project` — 任意のイベントでプロジェクトにアイテムを追加
- GraphQL API の `projectV2` — フィールド値の自動更新・アイテムの移動

参照: [Automating Projects using Actions](https://docs.github.com/en/issues/planning-and-tracking-with-projects/automating-your-project/automating-projects-using-actions)

#### Sub-issues（階層管理）

Issue の中に子 Issue を作る **Sub-issues** 機能を使うと、エピック → ストーリー → タスクのような階層構造を Projects 上で管理できます。Parent / Sub-issue フィールドで進捗が自動集計されます。

#### プロジェクトテンプレート（Organization 向け）

Organization の管理者は、チーム共通のフィールド・ビュー・Workflow を設定済みのテンプレートを作成できます。新プロジェクトのたびにゼロから設定する手間がなくなります。

---

### 3. 他のタスク管理ツールとの比較

「GitHub Projects で十分か、別ツールを検討すべきか」の判断材料を整理します。

| 観点 | GitHub Projects | Jira | Linear | Notion |
|------|----------------|------|--------|--------|
| GitHub との連携 | ネイティブ（最強） | プラグイン | 連携あり | 弱い |
| 導入コスト | 低い（GitHub があれば使える） | 高い | 中程度 | 低い |
| カスタマイズ性 | 中程度 | 高い | 中程度 | 高い |
| スプリント管理 | Iteration フィールドで可能 | 専用機能あり | 強い | 弱い |
| レポート・集計 | Insights（基本的） | 豊富 | 基本的 | 弱い |
| 向いているチーム | GitHub 中心の開発チーム | 大規模・複雑な組織 | スタートアップ | ドキュメント重視 |

**GitHub Projects で十分なケース**:
- 既に GitHub で Issue・PR を管理している
- ツールを増やしたくない
- チームが小〜中規模（10〜20 人程度まで）

**別ツールを検討するケース**:
- 複雑なレポートや SLA 管理が必要
- 開発以外の部署（営業・マーケ）と同じツールで管理したい
- カスタムワークフローが非常に複雑

---

### 4. 直感を育てるために

教材を読むだけでは身につきません。以下を意識しながら実際に使い続けてください。

- **まず小さく始める**: フィールドは Status だけ、ビューは 2 つだけでスタートし、「不便だな」と感じたときに追加する
- **スプリントごとに振り返る**: Insights の Burn up を見ながら「なぜ Done が伸びなかったか」を話し合う
- **うまくいったパターンをビューとして保存する**: 便利なフィルター・ソートを見つけたら名前をつけて保存しておく
