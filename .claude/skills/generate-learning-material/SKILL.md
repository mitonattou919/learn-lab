# 学習教材ジェネレーター

`docs/` に蓄積されている学習教材と同じスタイルで、新しいテーマの教材を作成します。**モデルの内蔵知識のみに頼らず、必ず一次情報をリサーチしてから書くこと**。

## 使い方

```
/generate-learning-material <テーマ> [tutorial|best-practices]
```

例:
- `/generate-learning-material Vitest tutorial`
- `/generate-learning-material Terraform best-practices`

## スタイルの参照基準（必ず冒頭で読むこと）

執筆を始める前に、以下のファイルを読んで構造・トーン・章立てのスタイルを取り込んでください。

- @docs/claude-code-tutorial.md — tutorial スタイルの理想形
- @docs/claude-code-best-practices.md — best-practices スタイルの理想形
- @docs/pester.md — 別ジャンルでの適用例
- @docs/google-adk-lesson.md — 別ジャンルでの適用例

## ワークフロー（5 フェーズ・対話型）

各フェーズが完了したらユーザーに見せて承認を取り、勝手に次へ進まないこと。

---

### Phase 1: ヒアリング

`AskUserQuestion` で以下を確定する：

1. **テーマ名**（コマンド引数で渡されてれば再確認程度）
2. **対象読者**: tutorial（未経験者向けハンズオン） / best-practices（中級者向け運用知識）
3. **出力ファイル名**: 既定 `docs/<theme>-tutorial.md` または `docs/<theme>-best-practices.md`
4. **一次情報のソース**:
   - 公式ドキュメント URL（必須レベルで推奨）
   - 公式リポジトリ URL
   - チュートリアル / ブログの URL
   - ローカル参照資料のパス
   - ユーザーが既に詳しい場合は「インタビュー駆動」も選択肢

URL/リポジトリが無いテーマは「内蔵知識のみで書くと古く・間違いやすい」と警告してから進めるか、ユーザーと一緒にソースを探す。

---

### Phase 2: 探索・リサーチ ⭐重要

**モデル知識だけで章立てを作らない**。必ず一次情報を取りに行く。

#### 並列リサーチ方針

`Agent` ツール（subagent_type=Explore または general-purpose）を **最大3並列** で起動して情報収集を委譲する。サブエージェントを使うことで、メイン会話のコンテキスト消費を抑える。

リサーチタスクの例：

| エージェント | 役割 |
|------------|------|
| 公式ドキュメント | `WebFetch` で公式サイトの主要ページを巡回し、章立てと核心概念を抽出 |
| 実践情報 | チュートリアル、ベストプラクティス記事、よくある質問を収集 |
| 失敗パターン | GitHub Issues、Stack Overflow、ブログから typical pitfalls を収集 |

各サブエージェントには「**500語以内の構造化サマリで返して**」と指示してコンテキストを節約する。

#### 単体ツールでのリサーチ

ローカル参照資料がある場合は `Read` `Glob` `Grep` を直接使う。短い WebFetch 1〜2回で済むなら直接呼んでもよい。

#### リサーチ結果のまとめ

集めた情報を **3〜5 項目の核心概念** と **想定される章候補 8〜10個** にまとめてユーザーに提示し、`AskUserQuestion` で「この方向で章立てに進む / 追加リサーチが必要 / 別ソースを当たる」を選ばせる。

---

### Phase 3: 章立て提案

リサーチ結果と下記の典型構造を踏まえ、章立て案を提示。`AskUserQuestion` で「このまま進める / 章を増減 / 大幅に作り直す」を選ばせる。

#### tutorial の典型章立て（8章前後）

```
Chapter 0: <テーマ> とは（概念と Why、従来との比較）
Chapter 1: インストールと初回ハロー体験（5分で成功体験）
Chapter 2: 基本操作
Chapter 3: 良い使い方の原則（指示・設計の作法）
Chapter 4: 段階的な進め方・失敗からの復帰
Chapter 5: 永続化・設定ファイル（あれば）
Chapter 6: 運用・管理
Chapter 7: 次の一歩（best-practices 文書への橋渡し）
```

#### best-practices の典型章立て（9章前後）

```
Chapter 0: <テーマ> とは（中核となる制約・トレードオフまで掘る）
Chapter 1: 環境構築と前提
Chapter 2: 中核原則（最重要パターン）
Chapter 3: 永続化・プロジェクト設定
Chapter 4: セッション/状態運用
Chapter 5: 拡張機構（プラグイン・サブシステム等、あれば）
Chapter 6: 自動化とスケール
Chapter 7: よくある失敗パターン
Chapter 8: チーム/業務統合
Chapter 9: 直感を育てる（締め）
```

テーマの性質に応じて柔軟に増減してよいが、**Chapter 0 = 概念と Why** と **最終章 = 次の一歩 / 直感** は必ず含めること。

---

### Phase 4: 章ごとに執筆

承認された章立てに沿って **1 章ずつ書き**、書き終わるたびにユーザーに見せて以下を聞く：

- このまま次へ進む
- この章を修正
- 章立て自体を見直す

複数章を一気に書かない。コンテキスト圧迫と方向ズレの両方を防ぐ。

#### Just-in-Time リサーチ

執筆中に Phase 2 の情報で足りないと判断したら、その場で **追加 WebFetch / サブエージェント** を起動してよい。むしろ推奨。具体的には：

- 章で扱う API/コマンドの正確な仕様確認
- バージョン依存の挙動の確認
- 公式の最新スクリーンショット/エラーメッセージの確認
- 「悪い例 → 良い例」の元ネタ収集

**捏造（ハルシネーション）したコマンド・API・URL を絶対に書かない**。確証が無いものは「未確認」と明示するか、ユーザーに確認する。

---

### Phase 5: 仕上げ

全章書き終わったら：

1. 章間の参照リンクを整える（「Chapter X 参照」など）
2. 末尾に **関連リソース** 節（公式ドキュメント URL、関連教材へのリンク）
3. tutorial なら練習課題が `<theme>-practice/` を章をまたいで使う設計になっているか確認
4. 兄弟教材（tutorial ⇄ best-practices）が両方ある場合は相互リンクを末尾に追加
5. **ルート `README.md` の `docs/` 一覧表に新教材の行を追加**（対象読者と一行説明を既存エントリのトーンに揃える）
6. 最終チェック: 文体統一、絵文字過剰、捏造の疑いを通読確認

---

## スタイルガイド（執筆時の遵守事項）

### 章の内部構造テンプレート

```
## Chapter N: <タイトル>

### 1. <導入セクション>
[Why から入る、または前章の復習]

### 2. <中核概念>
[説明 → 比較表 または Before/After]

### 3. <実践セクション>
[コード例 または ハンズオン手順]

### 4. <応用 / まとめ>
[選択基準表、テクニック集など]

### 練習課題（tutorial のみ必須）
1. 基本確認
2. 応用
3. 複合タスク
```

### トーンと体裁

| 項目 | ルール |
|------|--------|
| 文体 | 敬体（です・ます）で統一 |
| 絵文字 | ⭐（最重要）、⚠（注意）、✅❌（表内の○×）程度に限定。乱用しない |
| 比較表 | 概念導入の直後に「Before/After」または「<旧手段> vs <テーマ>」のような対比表を配置 |
| メタファー | 序盤で1つ用意（例: 「優秀なジュニアエンジニア」） |
| コード | 各セクションに最低1ブロック。`bash` `text` `markdown` 等を明示 |
| 引用ブロック | `> **うまくいかない時**:` のような実用的な脱線に使う |

### tutorial と best-practices の書き分け

| 観点 | tutorial | best-practices |
|------|----------|----------------|
| 前提知識 | ゼロから（インストールから） | 基本操作済みを仮定 |
| Chapter 0 | ツールの存在意義 | 中核制約・トレードオフ |
| 実装難度 | 5分で成功体験 → 単純タスク | 複合タスク・運用設計 |
| 失敗の扱い | エラー対処を都度挿入 | 失敗パターン集として独立章 |
| 練習課題 | 各章末に必須、`<theme>-practice/` を継続使用 | 不要（または巻末に1〜2問） |
| 章数 | 8章前後 | 9章前後 |
| 締め | 「次の一歩」で best-practices へ送る | 「直感を育てる」で締める |

---

## 出力先

既定: `docs/<theme>-<style>.md`

例:
- `docs/vitest-tutorial.md`
- `docs/terraform-best-practices.md`

ユーザーが Phase 1 で別パスを指定したらそれに従う。

---

## 重要な禁止事項

- スタイル参照基準のファイルを読まずに執筆を始めない
- **Phase 2 のリサーチを省略しない**。モデル内蔵知識のみで章立て・本文を作らない
- 捏造したコマンド・API・URL を書かない。不確実なものはリサーチで裏取り
- 章立ての承認を得る前に Phase 4 に進まない
- 1 章ずつ見せる原則を破って一気に複数章書かない
- 独自色を出しすぎず、既存教材のトーンに揃える
